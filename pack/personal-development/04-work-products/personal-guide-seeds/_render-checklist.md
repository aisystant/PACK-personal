# Чек-лист проверки рендера персонального руководства

> Канон проверки каждого рендера (дневного и недельного). Две части: содержательная
> (все входы генерации использованы) и организационно-техническая (формат выдачи,
> каскад Вход-Действие-Выход, процессы). Результат прогона - файл `checks/<дата>.md`
> в репозитории руководства пилота. Статусы: ✅ соблюдено · 🟡 частично · ❌ нарушено ·
> ⏳ утверждено, ещё не реализовано · ⬜ не применимо.

<details open>
<summary><b>Часть 1. Содержательная - входы генерации</b></summary>

> Инвариант для каждой строки: вход присутствовал при рендере, был свежим,
> и след его использования виден в выходе (файле руководства или табло).

| № | Вход | Производитель (откуда) | След в выходе |
|---|------|------------------------|---------------|
| 1.1 | Профиль развития: степень квалификации, ступень (для Ученика), срезы способностей | база платформы, срез Диагноста | степень по имени ведёт рассказ; данные совпадают во всех файлах |
| 1.2 | Узкое место (реальный слабый слот производного профиля) | производный профиль | тема или задание дня реагируют на слабый слот; расшифровка на табло |
| 1.3 | Квалификационная рамка: глубина, целевой объём, веса трёх программ (личное/рабочее/исследовательское развитие) | таблица весов по степеням | доли содержания по программам соответствуют степени |
| 1.4 | Заготовка квалификации (рамка, рычаги, мем, вопрос дня) | Пак, personal-guide-seeds | тема дня из рычагов; статус заготовки честно во frontmatter |
| 1.5 | Ступенно-доменные заготовки (для степени Ученик) | Пак, stage + domain | язык и примеры соответствуют ступени и домену |
| 1.6 | Снимок универсальных руководств (свежесть до 48 часов, фрагменты по программам) | ночной сбор с платформы | реальная выдержка в тексте + ссылка «читать целиком» на источник |
| 1.7 | Месячный стратегический контракт: цели месяца, неудовлетворённости, узкое место недели, привязки РП к целям | производный контракт стратегии | выбор темы отражает контракт; статус ok/stale/missing честный; дословных формулировок неудовлетворённостей в тексте нет |
| 1.8 | Связь неудовлетворённостей с узким местом | документ неудовлетворённостей | недельное «Зачем это тебе сейчас» - пересказом |
| 1.9 | Активные РП и план недели | реестр активных РП + недельный план | «На практике» - конкретный следующий шаг по приоритетному РП (приоритет по узкому месту недели, не по бюджету); повестка на табло актуальна |
| 1.10 | Тема месяца | файл темы месяца в репо | недельная тема согласуется с темой месяца |
| 1.11 | Активность за 7 дней + метрики среды | журнал событий | цифры только на табло («Статистика периода»); в руководстве - выводы словами |
| 1.12 | Рефлексии пилота за 7 дней (что узнал / что завтра) | history/ репо пилота | намерение пилота приоритизирует день; при отсутствии рефлексий - честная деградация, не выдумка |
| 1.13 | Вчерашний день (предыдущий фокус) | вчерашний файл руководства | сцепление: тема продвигается, не повторяется дословно |
| 1.14 | Выбор задания: степень × узкое место × домен + детектор пробелов | селектор заданий | задание на табло; совпадение с узким местом честно во frontmatter |
| 1.15 | Поли-контекст программ (если у пилота включён) | контекст мастерской | учтён без дублирования основной программы |
| 1.16 | Развитие вне явных РП | зарезервировано (решение пилота по охвату не принято) | пока честная пометка «не входит» |

</details>

<details>
<summary><b>Часть 2. Организационно-техническая</b></summary>

**А. Формат выдачи**

| № | Инвариант |
|---|-----------|
| 2.1 | Дневное руководство: шапка «Сегодня, дата», тема дня отдельным заголовком, шесть обязательных блоков (В одном предложении / Понятия / Объяснение / На практике / На что обратить внимание / Вопрос для рефлексии), блок «Вспомни понятия» последним |
| 2.2 | Недельное руководство: Кто ты сейчас → Тема недели → Опора (понятия + ссылки) → Четыре мастерства словами → Метод недели → Твои проекты → Задание недели → Ритм |
| 2.3 | Табло: «Где ты сейчас» (степень по имени + узкое место с расшифровкой) / «Статистика периода» / «Рабочая повестка» (актуальные РП, полные названия) / «Задание на сегодня» (одно, проверяемое) |
| 2.4 | Именование: степень и ступень только по имени; слово «ступень» у степеней выше Ученика не употребляется |
| 2.5 | Объём: не ниже целевого из шапки файла более чем на 15%; учебный текст насыщенный, без искусственного добива |
| 2.6 | Стиль: понятный русский; без английского жаргона, кодов методологии, служебных разделов и статистики в руководстве |
| 2.7 | Задания и цифры - только на табло; в руководстве разбор РП только в «На практике» (кратко - в «На что обратить внимание») |
| 2.8 | Все ссылки в теле резолвятся (проверка кодом); модель ссылки не сочиняет |
| 2.9 | Чувствительность: в публичном репо нет дословных неудовлетворённостей и данных семейного домена |

**Б. Каскад Вход-Действие-Выход (шесть принципов)**

| № | Принцип | Инвариант для руководства |
|---|---------|---------------------------|
| 2.10 | Триада | у дневного файла есть Вход (откуда тема), Действие (учебный текст), Выход (вопрос для рефлексии → history/) |
| 2.11 | Сцепление | выход вчерашнего дня стал входом сегодняшнего; рефлексия учтена следующим рендером |
| 2.12 | Инвариант входа | все входы либо производятся каскадом, либо помечены как внешние; стратегия - из производного контракта, не парсингом монолита |
| 2.13 | Трассируемость | у каждого блока есть производитель и потребитель; задание имеет место фиксации результата |
| 2.14 | Перенос незавершённого | непрочитанное/невыполненное не исчезает молча (дайджест, перенос) |
| 2.15 | Антипустышка | каждая секция трассируется к обещанию доставки руководства; секция без потребителя - кандидат на удаление |

**В. Процессы**

| № | Инвариант |
|---|-----------|
| 2.16 | Ночной рендер прошёл (дата, время, режим); в понедельник есть и дневной, и недельный |
| 2.17 | Прошлая неделя заархивирована в history/ при смене недели |
| 2.18 | Оглавление (guide.json) валидно: дни и неделя не смешаны, ссылки на существующие файлы |
| 2.19 | Стражи формы/содержания после сборки прошли; статус и находки зафиксированы |
| 2.20 | Флаги качества во frontmatter честные и полные (стратегический контекст, верность степени, статус заготовки, совпадение задания с узким местом) |
| 2.21 | Отпечаток конфигурации рендера присутствует (воспроизводимость) |
| 2.22 | Согласованность файлов: степень, ступень, узкое место одинаковы в руководстве, табло, портрете, README и оглавлении |
| 2.23 | Результат прогона записан в checks/<дата>.md с итоговым вердиктом |

</details>

## Формат результата прогона

Файл `checks/<дата>.md` в репозитории руководства: frontmatter (дата, проверенные файлы,
итоговый вердикт зелёный/жёлтый/красный, счётчики по статусам) + таблицы обеих частей
со статусом и комментарием по каждой строке + раздел «Находки» (дефекты с маршрутом:
кто чинит). Проверяющий - роль Верификатора (не автор рендера); допустим агентный
прогон с выборочной сверкой пилотом.
