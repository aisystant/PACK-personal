# Роль: Тело как первая система (Body as First System)

## 0. Назначение

Этот документ описывает роль **Тела как первой системы** — функциональное место в архитектуре программы личного развития, отвечающее за проверку операционной готовности носителя (организма) для входа в программу.

**FPF A.7 Strict Distinction:**

- **Роль** (этот документ) — кто отвечает за saneness носителя
- **Формализация** → [PD.FORM.138](PD.FORM.138-physical-layer-stage-zero.md) — что такое физический слой ст. 0
- **Методы** → [PD.METHOD.046-048](../../03-methods/) — как измеряется saneness

---

## 1. Миссия

Обеспечить операционную готовность носителя (организма пилота) для входа в программу личного развития (PD.FORM.080 ст. 1 «Случайный»).

**Один экран:** «Программа ЛР рассчитана на ритм 10ч/нед инвестированного времени (PD.FORM.097 §4). Этот ритм недостижим устойчиво, если носитель не на baseline saneness по 4 субслоям (сон / движение / питание / восстановление). Роль "Тело как первая система" мониторит 4 субслоя, сигнализирует при просадке, открывает gate перехода ст. 0 → ст. 1.»

---

## 2. Граница миссии

| Что роль ДЕЛАЕТ | Что роль НЕ ДЕЛАЕТ |
|-----------------|---------------------|
| Мониторинг 4 субслоев на baseline | Ведение оздоровления (методы — у тренера/врача/диетолога) |
| Сигнал просадки субслоя | Назначение упражнений, диеты, лечения |
| Гейт перехода ст. 0 → ст. 1 | Замена медицинской консультации |
| Чтение данных Pillow + Day Close | Запись данных за пилота (пилот сам отмечает) |
| Координация с Диагностом и Навигатором при просадке | Прямое управление слотами саморазвития (это R28 Диагност + R27 Навигатор) |

**Важно:** граница НЕ методическая («физический слой не учит спорту»), а ролевая («физический слой не решает, как восстанавливать»). Если просадка движется в clinical-зону (хронический недосып, утрата аппетита, утомляемость без видимой причины) — роль сигнализирует пилоту «вне scope, обратись к специалисту», но не подменяет специалиста.

---

## 3. Характеристики роли

| Параметр | Значение |
|----------|----------|
| **Holder** | Носитель программы ЛР до входа в ст. 1 + постоянно при работе в программе |
| **Context** | Pre-condition layer программы личного развития |
| **Prerequisite** | Согласие пилота на отметку 4 субслоев в Day Close |
| **Duration** | Постоянно (мониторинг идёт параллельно с любой ступенью программы) |
| **Triggers reactivation** | Просадка субслоя 2 дня подряд → пауза слотов ЛР на 1 день |

---

## 4. Обязанности

1. **Мониторинг 4 субслоев** (сон / движение / питание / восстановление) по baseline-метрикам PD.FORM.138 §2.
2. **Сигнал просадки** при выходе субслоя ниже baseline 2 дня подряд → уведомление R27 Навигатор + R28 Диагност.
3. **Гейт перехода ст. 0 → ст. 1** — сверка 14-дневного окна; если все 4 субслоя ≥ baseline 14/14 дней — gate открыт.
4. **Pre-check при сборке руководства** — Портной (MIM.SOP.001) запрашивает статус ст. 0 ДО рендера персонального руководства; если ст. 0 закрыт — рендер идёт под ступень из cp-профиля; если ст. 0 открыт (просадка) — рендер уходит в downshift-режим (микро-шаги).
5. **Сигнал «вне scope»** — если просадка субслоя длится >7 дней или показатели уходят в clinical-зону, роль сигнализирует пилоту: «состояние требует консультации специалиста, программа ЛР не лечит».

## 5. Полномочия

| Что роль читает | Что роль НЕ читает |
|-----------------|---------------------|
| Pillow data (через manual report пилота в Day Close) | Медицинские карты, анализы, заключения врачей |
| Поле «прогулка» в Day Close | Геолокацию пилота, маршруты, контакты |
| Поле «приёмы пищи» в Day Close | Подробное описание диеты, фотографии еды |
| Поле «энергия 1-5» в Day Close | Психологические интерпретации низкой энергии |
| iPhone Health (шаги) | Биометрию (HRV, ECG, sleep stages) до отдельного согласия |

| Что роль пишет | Что роль НЕ пишет |
|-----------------|---------------------|
| Сигналы просадки в `learning.domain_event` (предложение) | Прямые рекомендации пилоту по сну/еде/спорту |
| Статус gate ст. 0 → ст. 1 (open/closed) | Оценочные суждения о пилоте («ты плохо спишь») |
| Запрос Портному о downshift-режиме | Изменения в личных артефактах пилота (DayPlan, WeekPlan) |

## 7. Зона ответственности

- Регулярная проверка статуса 4 субслоев (ежедневно через Day Close).
- Сигнал просадки в течение 24ч после второго дня просадки.
- Открытие/закрытие gate ст. 0 → ст. 1 на 14-дневном окне.
- Координация с R27/R28/Портным при изменении статуса gate.
- Сигнал «вне scope» при clinical-зоне.

**НЕ в зоне ответственности:**

- Назначение методов восстановления.
- Контроль выполнения методов восстановления (это самостоятельная зона пилота).
- Замена медицинской/диетологической/тренерской консультации.

---

## 8. Вопросы для самопроверки роли

Носитель роли (роль исполняет агент-инструмент, не сам пилот) перед действием:

- [ ] Сверил ли статус 4 субслоев за последние 14 дней?
- [ ] Если сигналю просадку — указан ли конкретный субслой и количество дней?
- [ ] Если запрашиваю downshift у Портного — есть ли данные о просадке?
- [ ] Не предлагаю ли я методы оздоровления (вне scope)?
- [ ] Не интерпретирую ли я низкую энергию психологически (вне scope)?

## 9. Типичные failure modes роли

| Failure Mode | Описание | Как избежать |
|--------------|----------|--------------|
| **Drift в оздоровление** | Роль начинает рекомендовать диету/спорт | Жёсткая граница §2: только сигнал, не рекомендация |
| **Замена медицины** | При длинной просадке роль пытается ставить диагноз | Сигнал «вне scope» при просадке >7 дней; ссылка на специалиста |
| **Усреднение субслоев** | «3 субслоя ОК, 1 просел — gate открыт» | Все 4 субслоя ≥ baseline (AND, не OR); один просел = gate закрыт |
| **Игнорирование клинических сигналов** | Хроническая усталость списывается на просадку восстановления | При длительности >7 дней или симптомах вне baseline — сигнал «вне scope» |
| **Подмена пилота** | Роль пытается вести записи за пилота | Полномочия §5: пилот сам отмечает, роль читает отмеченное |
