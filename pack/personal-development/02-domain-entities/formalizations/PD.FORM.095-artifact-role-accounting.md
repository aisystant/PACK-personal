# Формализация: Учёт артефактов созидателя

## 1. Определение

**Артефакт созидателя** — любой материальный результат созидательной или учебной деятельности пользователя, зафиксированный в одном из его персональных репозиториев. Артефакт возникает в момент коммита (Git) или публикации и несёт в себе сигнал о том, в какой роли пользователь действовал в момент создания.

**Зачем учитывать артефакты:**

1. **RCS-сигнал.** Артефакты — наблюдаемые следы практики. Без них RCS (PD.FORM.089) опирается только на self-report, что ненадёжно. Коммит → неопровержимый факт: человек сделал что-то в этой роли в это время.
2. **Диагностика роста.** Портной (MIM.R.007) и Диагност (MIM.R.009) используют частоту и тип артефактов как один из источников для определения ступени и bottleneck.
3. **Питание Activity Hub.** Таблица `artifact_events` (#3 activity-hub или #11 learning — решается в WP-109/WP-253) пополняется событиями при каждом push, что позволяет строить timeline развития пользователя без опросов.

**Ограничение.** Количество артефактов ≠ качество развития. Артефакт — сигнал практики, не сигнал мастерства. Ступень определяется через RCS-профиль (FORM.089), а не через счётчик коммитов.

## 2. Четыре репозитория пользователя-созидателя

Пользователь действует в четырёх персональных репозиториях. Каждый репозиторий соответствует конкретному набору ролей (PD.FORM.087) и имеет определённый способ учёта.

| Репо пользователя | Что содержит | Роль исполнителя | Приватность | Способ учёта |
|---|---|---|---|---|
| `personal-strategy/` | Неудовлетворённости, стратегия, Lifework-пакет, планы, сессии стратегирования | Стратег (уровни охвата FORM.090, горизонт квартал+), Ученик (планирование недели) | Приватный | Коммиты в strategy-файлы, созданные/обновлённые документы планирования |
| `personal-knowledge-index/` | Личный блог, заметки, Песочница, Новая грамотность, черновики постов | Ученик (заметки, черновики) → Просветитель (опубликованный пост, ст. 5+) | Приватный до публикации | Коммиты md-файлов; публикация в клуб = отдельное событие |
| `PACK-personal/` (субсет через Gateway) | Паспорт домена, мировоззрение, личные показатели, RCS, характеристики | Исследователь (создаёт новые форматы, методы), Ученик (обновляет собственные данные) | Платформенный L2, пользователь видит субсет | Изменения через Gateway personal_write + коммиты в платформенный PACK |
| `personal-guide/` (выдача Портного) | Персональные руководства, ДЗ-пакеты, книга ЛР v3, weekly/daily | Ученик (получатель контента, выполняет ДЗ) | Приватный | Выполнение ДЗ-пакетов (специфические коммиты), отметки прохождения |

**Правило репозитория как контекста.** Репо не определяет роль однозначно — роль определяет **тип артефакта** внутри репо. Заметка в `personal-knowledge-index/` = Ученик; опубликованный пост оттуда = Просветитель. Стратегический план в `personal-strategy/` = Стратег; ДЗ-шаблон недели там же = Ученик.

## 3. Классификация по ролям исполнителя

Каждый артефакт несёт сигнал о роли, в которой пользователь действовал. Роли берутся из ролевой траектории PD.FORM.087.

### 3.1 Схема: тип артефакта → роль

| Тип артефакта | Признаки | Роль исполнителя | Ступень |
|---|---|---|---|
| Выполненное ДЗ | Файл в `exercises/`, `homework/`, ответ на ДЗ-пакет, отметка в `personal-guide/` | Ученик | Любая (ст. 1-5) |
| Заметки, черновики | Fleeting notes, черновики постов, Песочница, незаконченный блог | Ученик | Любая (ст. 1-5) |
| Личный стратегический документ | Цели, Lifework, неудовлетворённости, стратегия квартала/года, системный разбор | Стратег / Ученик (ст. 3+) | ст. 3+ |
| Публикация | Пост в клуб/социальные сети с завершённым нарративом, опубликованная статья | Просветитель | ст. 5 |
| Предметная разработка в Pack | Новая формализация (PD.FORM.*), метод (PD.METHOD.*), SoTA | Исследователь | ст. 4+ |
| Недельный / дневной план | WeekPlan, DayPlan, гипотеза роста | Ученик (планировщик) | Любая (ст. 1-5) |
| Ревью и ретроспектива | Week Close, Day Close, Month Close отчёты | Ученик (рефлексирующий) | ст. 2+ |
| Системный разбор | Анализ ситуации с применением системного мышления (M4), ArchGate | Интеллектуал / Исследователь | ст. 4+ |

### 3.2 Приоритет классификации

При пограничных случаях (артефакт несёт признаки нескольких ролей):

1. **Публикация** перевешивает заметку — если заметка опубликована, роль = Просветитель.
2. **Применение метода** перевешивает выполнение задания — если человек применяет системный разбор к своей задаче, роль = Интеллектуал (не Ученик).
3. **Новый артефакт Pack** перевешивает личный документ — создание PD.FORM.NNN = Исследователь независимо от репо.

## 4. Event Model: структура таблицы artifact_events

Каждый артефакт фиксируется одним событием в таблице учёта. Таблица живёт в БД #3 (activity-hub) или #11 (learning) — решается ArchGate в WP-109/WP-253.

```
artifact_events
─────────────────────────────────────────────────────────────────────
artifact_id        UUID, PK — уникальный идентификатор события
user_id            UUID (ory_id) — идентификатор пользователя в Ory
repo               ENUM — один из 4 репо (personal-strategy,
                   personal-knowledge-index, pack-personal, personal-guide)
file_path          TEXT — относительный путь к файлу в репо
artifact_type      ENUM — тип артефакта (homework, note, draft,
                   strategy-doc, publication, pack-entity,
                   weekly-plan, daily-plan, review, system-analysis)
role               ENUM — роль исполнителя (learner, strategist,
                   intellectual, researcher, enlightener)
stage_at_creation  INT (1-5) — ступень пользователя в момент создания
                   (из текущего RCS.stage, кеш из Память.Derived)
w_weight           FLOAT [0,1] — вес вклада в срез Мировоззрение
m1_weight          FLOAT [0,1] — вес вклада в срез M1 (методы саморазвития)
m2_weight          FLOAT [0,1] — вес вклада в срез M2 (IWE/ОРЗ)
m3_weight          FLOAT [0,1] — вес вклада в срез M3 (предметный домен)
m4_weight          FLOAT [0,1] — вес вклада в срез M4 (системное мышление)
classifier_source  ENUM — кто классифицировал (auto-haiku, manual,
                   auto-confirmed — авто с ручным подтверждением)
confidence         FLOAT [0,1] — уверенность классификатора
created_at         TIMESTAMPTZ — момент создания артефакта (Git commit time)
indexed_at         TIMESTAMPTZ — момент записи в таблицу (при обработке push)
```

**Веса срезов** (`w_weight`, `m1_weight`, ...) — не бинарные (0/1), а непрерывные. Один артефакт может вносить сигнал сразу в несколько срезов (например, системный разбор своей недели = M4 + M1). Сумма весов не обязана равняться 1.0. Нормировка — на уровне агрегата при расчёте RCS.

**stage_at_creation** кешируется из Память.Derived в момент записи события. Это не «правильная» ступень — это ступень, которая была видна платформе в момент создания артефакта. Служит для исторических срезов: «как выглядела практика на ступени 2 у этого пользователя».

## 5. Классификатор: алгоритм автоматической классификации

Классификация происходит в момент обработки Git push-события (webhook или pull-based polling).

### 5.1 Алгоритм (три шага)

```
ШАГ 1. Правила-эвристики (детерминированные, без LLM)
────────────────────────────────────────────────────────
Применяется по пути файла и имени:

IF path matches "exercises/|homework/|personal-guide/**/*.md" → type=homework, role=learner
IF path matches "WeekPlan|DayPlan|weekly/|daily/" → type=weekly-plan/daily-plan, role=learner
IF path matches "Week Close|Day Close|Month Close|*-close-*" → type=review, role=learner
IF path matches "PD.FORM.*|PD.METHOD.*|PD.SOTA.*" → type=pack-entity, role=researcher
IF path in repo=personal-strategy AND matches "strategy|lifework|dissatisfaction" → type=strategy-doc, role=strategist

Если правило сработало с confidence=0.95 → записать, classifier_source=auto-haiku, пропустить шаг 2.

ШАГ 2. Субагент-классификатор (Haiku, context-isolated)
─────────────────────────────────────────────────────────
Вход: path + первые 200 символов содержимого + diff-stat.
Промпт: «Определи тип артефакта и роль исполнителя из следующего набора (ENUM выше).
         Верни JSON: {type, role, w, m1, m2, m3, m4, confidence}.»
Выход: JSON с типом, ролью и весами срезов.
Classifier_source = "auto-haiku".

ШАГ 3. Ручная верификация пограничных случаев
──────────────────────────────────────────────
Триггер: confidence < 0.7 ИЛИ тип = publication (публикация всегда верифицируется).
Платформа добавляет события в очередь pending-review (Activity Hub).
Пользователь видит в `/progress` блок «Уточните» — подтверждает или корректирует.
После ручного решения: classifier_source = "manual" или "auto-confirmed".
```

### 5.2 Таблица весов по умолчанию (стартовая конфигурация)

Если Haiku не вернул явные веса или confidence < 0.5, используются веса по типу:

| Тип артефакта | W | M1 | M2 | M3 | M4 |
|---|---|---|---|---|---|
| homework | 0.0 | 0.5 | 0.3 | 0.2 | 0.0 |
| note / draft | 0.1 | 0.3 | 0.2 | 0.3 | 0.1 |
| strategy-doc | 0.3 | 0.1 | 0.1 | 0.2 | 0.4 |
| publication | 0.2 | 0.1 | 0.1 | 0.5 | 0.2 |
| pack-entity | 0.2 | 0.0 | 0.1 | 0.3 | 0.5 |
| weekly-plan / daily-plan | 0.1 | 0.4 | 0.4 | 0.1 | 0.0 |
| review | 0.2 | 0.5 | 0.2 | 0.0 | 0.2 |
| system-analysis | 0.3 | 0.0 | 0.0 | 0.2 | 0.6 |

Веса уточняются эмпирически на реальных данных пилотов (WP-109, WP-253).

## 6. Применение в RCS: какой артефакт питает какой срез

Каждое событие `artifact_events` через веса `(w|m1|m2|m3|m4)_weight` вносит вклад в расчёт активности по срезам RCS (PD.FORM.089).

### 6.1 Логика агрегации

```
Для каждого среза S ∈ {W, M1, M2, M3, M4}:
    activity_S(period) = SUM( artifact_events.S_weight ) за период
    baseline_S = EWA( activity_S, alpha=0.3 )   // экспоненциально-взвешенное среднее
```

Полученные `baseline_S` дополняют (не заменяют) RCS baseline из self-report анкет (FORM.089 §4). Соотношение: авто-сигнал из артефактов = 40% веса; self-report = 60% веса. Пропорция пересматривается при наличии данных пилотов.

### 6.2 Какой тип артефакта важен для какого среза

| Срез RCS | Что ищем в артефактах | Сигнал |
|---|---|---|
| **W** (Мировоззрение) | Стратегические документы, системный разбор, публикации с концептуальным содержанием | Человек думает о большом — ступень 3+ |
| **M1** (Методы саморазвития) | Weekly/Daily планы, ревью, выполненные ДЗ, записи о ритме | Человек практикует саморегуляцию регулярно |
| **M2** (IWE / ОРЗ) | Коммиты в IWE-файлы, обновление личных репо, работа с экзокортексом | Человек использует среду как инструмент развития |
| **M3** (Предметный домен) | Заметки и публикации по домену, профессиональные артефакты, case studies | Ученик работает с предметом, не только с методами |
| **M4** (Системное мышление) | Pack-сущности, ArchGate, системный разбор ситуаций, Lifework-документы | Человек применяет системную оптику к своей деятельности |

### 6.3 Пороги активности как дополнительный сигнал диагноста

Диагност (MIM.R.009) может использовать следующие пороги при оценке ступени:

| Сигнал | Значение |
|---|---|
| 0 артефактов M1 за 2 недели | Ступень не выше 1 (Случайный) — нет регулярной практики |
| ≥3 weekly-plan за месяц | Есть признак ст. 2 (Практикующий) — регулярное планирование |
| ≥1 strategy-doc за квартал | Признак ст. 3+ (Систематический) — начало стратегического горизонта |
| ≥1 pack-entity или system-analysis за квартал | Признак ст. 4 (Дисциплинированный) — работа с M4 |
| ≥1 publication + ≥3 pack-entity за квартал | Признак ст. 5 (Проактивный) — созидание и передача |

Пороги предварительные — требуют калибровки на реальных данных пилотов.

## 7. Ключевое различение: учебный артефакт vs созидательный артефакт

Программа ЛР работает с двумя принципиально разными классами артефактов. Путать их = неверно оценивать ступень.

| | Учебный артефакт | Созидательный артефакт |
|---|---|---|
| **Что это** | Результат учебного процесса: применение метода для освоения навыка | Результат созидательной деятельности: продукт, создающий ценность для других |
| **Для кого** | Для самого пользователя (как ученика) | Для аудитории, системы, сообщества |
| **Примеры** | Выполненное ДЗ, заметка, черновик, дневник практики, недельный план | Публикация в клуб, написанная статья, созданный метод в Pack, системный разбор для команды |
| **Роль исполнителя** | Ученик (всегда) | Просветитель, Исследователь, Профессионал, Стратег |
| **Ступень, с которой возникает** | Ст. 1 (любой может делать ДЗ) | Ст. 3-4 для системных разборов; ст. 5 для публикаций |
| **Сигнал для RCS** | Активность в M1/M2 срезах | Активность в W/M4/M3 срезах |
| **Как учитывается** | По факту создания файла в репо | По факту публикации/завершения; пограничное — ручная верификация |

**Тест:** «Мог бы этот артефакт помочь кому-то кроме меня?» Да → созидательный. Нет → учебный.

**Антипаттерн.** Считать опубликованный пост «заметкой в репо» — значит не видеть, что человек перешёл в роль Просветителя. Аналогично: считать выполненное ДЗ «созидательным артефактом» — значит завышать ступень при диагностике.
