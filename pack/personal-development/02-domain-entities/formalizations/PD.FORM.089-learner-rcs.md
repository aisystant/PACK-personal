# Формализация: Role Characterisation Space (RCS) Ученика

## 1. Назначение

**Проблема.** PD.FORM.003 и PD.FORM.080 описывают ступени Ученика (1 Случайный → 5 Проактивный) как плоские нарративные состояния. При диагностике живого человека Диагност (MIM.R.009) и Навигатор (MIM.R.007) принимают решение «ступень = N» на основе смешанных сигналов: ритм, методы, мировоззрение, агентность, инструменты. Решение не воспроизводимо — два диагноста могут прийти к разным ступеням для одного человека.

**Решение.** Перейти от плоской ступени к **многомерному профилю слотов** по формализму FPF A.19 `U.CharacteristicSpace`. Каждый слот — отдельная характеристика с собственным показателем, значением и потенциалом (PD.FORM.088 §2). Ступень вычисляется из профиля явным правилом (§5), а не интуицией диагноста.

**Три эффекта:**

1. **Диагностика воспроизводима.** Алгоритм: измерить 7 слотов → применить правило → получить ступень + bottleneck. Два диагноста получают одинаковый результат.
2. **Портной видит, что тянуть.** Bottleneck (слабое звено) определяет фокус на квартал; ведущие слоты — на удержание baseline.
3. **Двойной gate Д→П** (FORM.080 §7.1) выражается формально: мировоззренческий слот и мастерственные слоты проверяются независимо.

**Не отменяет PD.FORM.003/080.** Нарративы ступеней остаются — RCS даёт им операционный низ: «как измерить, что человек на ступени 3».

## 2. Структура RCS: 7 слотов

> **Принцип выбора:** слоты соответствуют двум сквозным линиям PD.FORM.080 §3.1 (Мировоззрение × Мастерство M1-M4) + два независимых измерения (IT-level для канала, Agency для проактивности). Пересечений между слотами минимум — это **ортогональные оси**, а не каталог характеристик.

| # | Слот | Что это | Источник нарратива |
|---|------|---------|--------------------|
| **W** | **Мировоззрение** | Зрелость взгляда на себя/окружение/мир как систему | FORM.080 §3 (мировоззренческая дуга) |
| **M1** | **Собранность** | Управление вниманием, регулярность, ритуалы | FORM.080 §3.1 M1; FORM.038 (продуктивное состояние) |
| **M2** | **IWE/ОРЗ** | Владение экзокортексом и протоколами Открытия-Работы-Закрытия | FORM.080 §3.1 M2 |
| **M3** | **Проф-методы** | Владение SoTA-методами домена (опционально) | FORM.080 §3.1 M3 |
| **M4** | **Системное мышление** | Системный взгляд и системный разбор (роли, границы, эмерджентность) | FORM.080 §3.1 M4; FORM.087 (траектория) |
| **IT** | **IT-level** | Владение цифровыми инструментами (бот/VS Code/Claude Code/MCP) | Определяет канал доставки |
| **A** | **Agency** | Способность инициировать изменения, действовать автономно | FORM.087 (Проактивный ученик) |

**Статус слотов:**
- **Стержневые** (входят в вычисление ступени, §5): W, M1, M2, M4
- **Вспомогательные** (профиль, но не ступень): M3 (домен-зависимый), IT (канал), A (gate на ст. 5)

### 2.1 Карта характеристик по осям

> **Назначение.** Сводная карта: какие характеристики созидателя относятся к оси Мировоззрение, какие к оси Мастерство. Используется Аттестатором (WP-310) и Симулятором (WP-310 Ф-Е). Механизм выбора осей: информация→знания→мировоззрение / методы в реальной жизни→мастерство (FORM.080 §3.1).

Полная цепочка измерений: **Ось → Характеристика → Показатель → Действие (событие) → Норматив по ступеням**.
Детализация показателей и действий: §12 (авто-расчёт Аттестатора) + FORM.093 §3-§5 (характеристики созидателя на ступени 5).

**Ось Мировоззрение** — как созидатель видит себя, окружение и мир как системы:

| Характеристика | Показатель | Документ |
|----------------|-----------|----------|
| Зрелость мировоззрения | `W.baseline_idx` (фаза мировоззренческой дуги 1-5) | §4 W |
| Агентность | `A.baseline_idx` (доля самоинициированных действий) | §4 A |
| Системность мировоззрения *(авто)* | `worldview_score` (взвешенная сумма событий рефлексии) | §12 |
| Агентность *(авто)* | `agency_score` | §12 |
| Ясность | `clarity_idx` (различения в артефактах + self-report) | FORM.093 §3 #1 |

**Ось Мастерство** — что умеет делать: методы, инструменты, практика во всех доменах (включая организм и окружение):

| Характеристика | Показатель | Документ |
|----------------|-----------|----------|
| Собранность — методы M1 | `M1.baseline_idx` (ч/нед по методам) | §4 M1 |
| IWE/ОРЗ — инструмент M2 | `M2.baseline_idx` (доля задач через ОРЗ) | §4 M2 |
| Проф-методы — M3 (опционально) | `M3.baseline_idx` | §4 M3 |
| Системное мышление — M4 | `M4.baseline_idx` (применение различений) | §4 M4 |
| Систематичность *(авто)* | `self_dev_days_per_week` | §12 |
| Инвестированное время *(авто)* | `avg_hours_per_week` | §12 |
| Методичность мышления *(авто)* | `methodical_events_per_month` | §12 |
| Собранность (наблюдаемая) | `composure_idx` | FORM.093 §3 #3 |
| Регулярность | `regularity_idx` | FORM.093 §3 #4 |
| Способность производить | `production_capacity_idx` | FORM.093 §3 #5 |

**Нейтральная (канал, не ось развития):** IT-level (`IT.baseline_idx`) — определяет канал доставки, не ступень.

**Обе оси — условия устойчивости:** Продуктивность, Ресурсность, Стрессоустойчивость (FORM.093 §3 #6-8) — не входят в gate ступеней 1-4, проверяются на gate ступени 5.

> **Пометка *(авто)*.** Характеристики с этой пометкой вычисляются Аттестатором автоматически по событиям Activity Hub. Остальные требуют диалога (Диагност R28) или self-report.

## 3. Квартет каждого слота (по PD.FORM.088 §2)

Каждый слот = четвёрка:

| Элемент | Что фиксируется |
|---------|-----------------|
| **Характеристика** | Свойство системы-человека как аспект внимания |
| **Показатель** | Выбранная шкала и единица измерения |
| **Значение** | Два числа: **сейчас** (моментальное) и **baseline** (устойчивый уровень, сглаживание 4-8 недель) |
| **Потенциал** | Текущий потолок шкалы для данного человека |

**Правило формулировки цели:** любая цель программы ЛР в рамках слота обязана называть, что двигаем:
- «Поднять **baseline** слота X с N до M за квартал» → M1/M2 (методы)
- «Расширить **потенциал** слота X с N до M за год» → M4 (фундаментальное развитие)

Цель, сформулированная через «значение-сейчас», отбрасывается как реакция на шум (PD.FORM.088 §2.1).

## 4. Определения слотов (MVP-версия, уточняется в PD.FORM.089.v2)

> **Ограничение MVP:** формализм квартета заложен, конкретные шкалы и пороги — draft. Финальная версия со всеми шкалами, тестовыми заданиями и алгоритмом измерения выходит в следующей итерации (см. §9 Roadmap).

### W. Мировоззрение

| Элемент | Значение |
|---------|----------|
| Характеристика | Зрелость системного мировоззрения |
| Показатель | Фаза мировоззренческой дуги (FORM.080 §3), индекс 1-5 |
| Значение (сейчас) | Текущая фаза, определённая по ответам в диагностическом диалоге |
| Значение (baseline) | Фаза, устойчиво проявляющаяся 4+ недели в артефактах (записи, решения, постановка задач) |
| Потенциал | 5 (системное мировоззрение + agency) — для человека верхняя фаза дуги |

**Индекс фаз:**
1. Нет осознания — человек не видит себя как систему
2. «Я — система» — видит собранность, ресурс, методы (FORM.080 §3 стр.51)
3. «Окружение влияет на меня» — видит среду (FORM.080 стр.52)
4. «Мир — система, и я в ней деятель» — видит роли, влияния (FORM.080 §7.1)
5. Системное мировоззрение + agency — интегрированное, передача культуры

**Рубрики baseline-индексов 1-5.** Рубрика назначается по устойчивым признакам 4+ недель (не по одному ответу в диалоге).

| Индекс | Операционные признаки baseline |
|--------|--------------------------------|
| **1** | Проблемы объясняет внешними обстоятельствами («так сложилось», «характер такой»). Нет языка системы/метода. Артефакты: отсутствуют или чисто событийные. |
| **2** | Видит внутренние ресурсы: внимание, энергия, ритм. Различает «состояние» и «задачу». Пробует методы (учёт времени, помидорки). Артефакты: начатый дневник или журнал времени. В речи: «я собран/не собран», «у меня не было ресурса». |
| **3** | Замечает влияние среды (люди, мемы, пространство). Избирательно выбирает окружение, отказывается от токсичного. В речи: «окружение тянет», «я попал под влияние». Артефакты: записи о влиянии среды, осознанная настройка окружения (цифровое/физическое). |
| **4** | Описывает задачи через роли, границы, интерфейсы. Различает «я» и «моя роль». Видит институты как системы. В речи: «я как [роль]», «обещание системы», «граница ответственности». Артефакты: системные разборы ситуаций, явные границы проектов. |
| **5** | Интегрированное системное мировоззрение + agency. Создаёт культурные формы (методы, тексты, проекты), втягивающие других. В речи: «мы как система», «в нашей роли». Артефакты: наставничество, публикации, инициативы, обучающие других системному взгляду. |

### M1. Собранность

| Элемент | Значение |
|---------|----------|
| Характеристика | Управление вниманием, регулярность работы по методам |
| Показатель | Слоты работы по методам в неделю (помидорки / ОРЗ-сессии) |
| Значение (сейчас) | Фактические слоты текущей недели |
| Значение (baseline) | Скользящее среднее 4-8 недель |
| Потенциал | 10+ часов/нед как стиль жизни (FORM.003 ст. 5) |

**Якоря baseline по ступеням (FORM.003 §3):** 2 / 5 / 6 / 8 / 10 ч/нед для ст. 1-5.

**Рубрики baseline-индексов 1-5.** Baseline = скользящее среднее 4-8 недель. Всплеск в одной неделе не поднимает baseline.

| Индекс | Якорь ч/нед | Операционные признаки baseline |
|--------|-------------|--------------------------------|
| **1** | ~2 | Случайные всплески работы. Нет slot-планирования. Работа по срочности и настроению. Помидорки не используются или 1-2 за неделю. Срывы длятся месяцами. |
| **2** | ~5 | Практикует помидорки/слоты эпизодически. Минимум 1 регулярный слот в неделю. Держит ритм 2-3 недели, потом срыв. Восстановление после срыва — 2-3 недели. |
| **3** | ~6 | Устойчивый недельный ритм с короткими срывами. Помидорки как привычка (не усилие воли каждый раз). 3+ регулярных слотов. Восстановление после срыва — 1 неделя. |
| **4** | ~8 | Высокая регулярность, срывы краткие (<3 дней). Помидорки — стиль работы. Утренняя гигиена. Метод применяется без напоминаний. Планирование слотов встроено в Day Open. |
| **5** | ~10+ | Собранность как стиль жизни. 10+ часов без усилия воли. Ритм адаптируется к контексту (путешествия, болезнь, кризис) — не разрушается. Способен передать метод другому и поддерживать его. |

### M2. IWE/ОРЗ

| Элемент | Значение |
|---------|----------|
| Характеристика | Владение экзокортексом и протоколами ОРЗ |
| Показатель | Доля значимых задач (≥15 мин), прошедших через ОРЗ-ритуал; наличие структур Pack/DS |
| Значение (сейчас) | % задач за неделю |
| Значение (baseline) | Устойчивая доля + наличие зрелых артефактов (WP-контексты, captures, Note-Review) |
| Потенциал | 100% (все значимые задачи — через ОРЗ) + полный экзокортекс (Pack + DS) |

**Рубрики baseline-индексов 1-5.** Аудит по артефактам (Pack-репо, DS-репо, наличие WP-контекстов, следы Week/Month Close) + самоотчёт о доле задач через ОРЗ.

| Индекс | Операционные признаки baseline |
|--------|--------------------------------|
| **1** | ОРЗ не применяется. Задачи без явного Открытия и Закрытия. Экзокортекса нет (возможно, есть бот как канал). Артефакты случайны и не связаны. Нет WP-контекстов. |
| **2** | Open/Close сессии применяются эпизодически (<30% значимых задач). Начата Pack- или DS-структура, но слабо заполнена. 1-2 WP-контекста. Day Open/Close — не регулярно. Week Close — нет. |
| **3** | ОРЗ на значимых задачах (>60%). Day Open/Close — регулярно. Pack и DS структурированы по шаблонам (папки, CLAUDE.md, navigation). 5+ WP-контекстов. Week Close — регулярно. Memory-файлы обновляются. |
| **4** | ОРЗ как норма (>90% значимых задач). Протоколы работают «на автомате». Зрелый экзокортекс: Pack с 10+ формализациями, DS с чётким routing, субагенты применяются. **Month Close как ритуал с явным циклом обратной связи: его архив читается на ближайшей Strategy Session при пересмотре фокуса** (см. PD.METHOD.008 § Каскад ВДВ Stage 2). Трекер мультипликатора по месяцам. Расширения протоколов (`extensions/`). |
| **5** | 100% значимых задач через ОРЗ. Свои расширения протоколов, написанные и документированные для других. Передаёт метод (обучает/документирует). Экзокортекс — полная инфраструктура: Pack, DS, FMT-шаблоны, staging-канал. |

### M3. Проф-методы (опционально)

| Элемент | Значение |
|---------|----------|
| Характеристика | Владение SoTA-методами своего домена |
| Показатель | Число ключевых методов домена, встроенных в работу |
| Значение (сейчас) | Применяет ли метод в задачах домена |
| Значение (baseline) | Устойчивое применение + распознавание граничных случаев |
| Потенциал | Определяется SoTA домена (не фикс — расширяется с эволюцией SoTA) |

**Примечание:** не основная часть ЛР. Фокус появляется на ступени 5 (Проактивный) и в программе «Рабочее развитие». Полные домен-специфичные рубрики — в v4 (после проектирования программы «Рабочее развитие»).

**Рубрики baseline-индексов 1-5 (минимальная версия, домен-независимая).**

| Индекс | Операционные признаки baseline |
|--------|--------------------------------|
| **1** | SoTA домена не изучается систематически. Методы берёт случайно (что попалось). Нет устойчивого применения ни одного метода. |
| **2** | Знаком с 2-3 ключевыми методами домена. Применяет их в типовых задачах. Не знает границ применимости — использует «по умолчанию» во всех ситуациях. |
| **3** | Применяет 4-6 SoTA-методов домена. Знает, когда какой метод подходит, а когда нет. Замечает несоответствие метода задаче и переключается. |
| **4** | Полный рабочий стек SoTA-методов домена. Самостоятельно выбирает метод под новую задачу без подсказки. Отслеживает обновление SoTA, обновляет практику. |
| **5** | Создаёт или адаптирует методы под нестандартные задачи. Обучает других методам. Отслеживает горизонт SoTA и вносит вклад в его развитие (публикации, воркшопы, внутренние стандарты). |

### M4. Системное мышление

| Элемент | Значение |
|---------|----------|
| Характеристика | Способность к системному взгляду и системному разбору |
| Показатель | Применение различений (роль, система, граница, эмерджентность, деятельность) в разборе реальных ситуаций |
| Значение (сейчас) | Сколько различений применил в недельной порции задач |
| Значение (baseline) | Стабильно применяемые различения |
| Потенциал | Интегративная сложность + калибр задач, который человек способен удержать (PD.FORM.088 §4) — расширяется фундаментальным развитием |

**Рубрики baseline-индексов 1-5.** Измеряется разбором реальных ситуаций (не тест-викториной). Набор различений-опор: система/процесс, роль/исполнитель, граница, интерфейс, эмерджентность, обещание, калибр, деятельность.

| Индекс | Операционные признаки baseline |
|--------|--------------------------------|
| **1** | Различения не применяются. Язык бытовой, проблемы объясняются нарративами («плохой менеджер», «неудачный проект»). Путает систему с группой людей, роль с должностью. |
| **2** | Применяет 1-2 базовых различения (система vs процесс, метод vs инструмент). Частые ошибки. Начинает различать роли от исполнителей в знакомых контекстах. В незнакомой области — беспомощен. |
| **3** | Применяет 5-7 различений стабильно в знакомых контекстах. Находит роли, границы, интерфейсы. В новом контексте (чужой домен) — требуется помощь или образец. Отличает case от generic на примерах. |
| **4** | Применяет 10+ различений в разборе новых ситуаций без образца. Самостоятельно находит неочевидные системы (институт, метаинфраструктура, надсистема). Конструктивно критикует чужие разборы: показывает упущенные границы или роли. |
| **5** | Интегрированное системное мышление. Создаёт новые различения и обосновывает их через различение, тест, примеры. Удерживает калибр больших систем (организация, рынок, общество). Передаёт метод — обучает системному разбору других. |

### IT. IT-level

| Элемент | Значение |
|---------|----------|
| Характеристика | Владение цифровыми инструментами |
| Показатель | Уровень канала (0-5): нет / TG-бот / Web / VS Code + Claude Code / Claude Code + MCP + Git flow / full-stack CLI |
| Значение (сейчас) | Какой канал пользователь реально использует ежедневно |
| Значение (baseline) | Устойчивый уровень + эффективность использования |
| Потенциал | 5 (полный стек авторского IWE) |

**Независимая ось.** Не влияет на ступень Ученика, но определяет **канал доставки контента** Портным и Оркестратором.

**Рубрики baseline-индексов 0-5.**

| Индекс | Канал | Операционные признаки baseline |
|--------|-------|--------------------------------|
| **0** | Нет цифровых | Не использует цифровые инструменты для IWE/обучения. Только бумага, устное взаимодействие. |
| **1** | TG-бот | Работает с IWE через Telegram (aist_me_bot). Получает контент в чате, отвечает на вопросы, но ничего не устанавливает. |
| **2** | Веб | Использует claude.ai, aisystant.com через браузер. Загружает/скачивает файлы. Возможно Obsidian или Notion через веб. |
| **3** | VS Code + Claude Code | Установлен VS Code и Claude Code CLI. Открывает файлы проектов, делает коммиты (GUI или командная строка). Работает в существующем IWE. |
| **4** | Claude Code + MCP + Git flow | Активный пользователь IWE: ведёт Pack и DS репо, применяет MCP-инструменты (personal_write, knowledge_search и др.), понимает структуру CLAUDE.md и routing-правила. |
| **5** | Full-stack | Пишет хуки, скрипты, расширения (`extensions/`). Создаёт собственные MCP-серверы или скиллы. Настраивает CI/CD, launchd, cron для собственных ритуалов. |

### A. Agency

| Элемент | Значение |
|---------|----------|
| Характеристика | Способность инициировать изменения, действовать автономно |
| Показатель | Доля самоинициированных действий (проекты, экспедиции, эксперименты) vs реактивные (задания от окружения) |
| Значение (сейчас) | % за неделю |
| Значение (baseline) | Устойчивая доля + наличие собственных проектов (не заимствованных) |
| Потенциал | ≥80% самоинициированных (Проактивный, FORM.087) |

**Независимая ось.** Gate для перехода на ступень 5 — без agency человек остаётся на ступени 4, даже если все мастерственные слоты достигли baseline.

**Рубрики baseline-индексов 1-5.**

| Индекс | Операционные признаки baseline |
|--------|--------------------------------|
| **1** | Действует почти исключительно по запросу. Собственных проектов нет или они зависли на старте. Решения откладываются — ждёт разрешения, гарантий или чужой инициативы. В речи: «мне сказали», «когда можно будет», «не знаю, с чего начать». |
| **2** | Есть 1-2 личных намерения, но большинство задач — от окружения. Инициирует мелкие изменения в быту (переставить расписание, попробовать новый инструмент). Решения принимает медленно, часто запрашивает одобрение. Откладывает всё, где нет чёткого «правильного» пути. |
| **3** | ~50% задач из собственных целей. Держит один активный личный проект. Принимает решения в знакомых контекстах без чужого подтверждения. Готов ошибиться. В незнакомом контексте — нужен образец или поддержка. |
| **4** | >60% действий самоинициированы. Несколько независимых проектов одновременно. Действует в неопределённости: ставит эксперимент, оценивает результат, корректирует без потери темпа. Не ждёт разрешения для значимых личных решений. |
| **5** | >80% действий собственные. Формирует повестку для ближнего круга. Инициирует изменения в системах без формальной власти (команда, сообщество, организация). Создаёт условия, в которых другие становятся проактивными. В речи: «я решил попробовать», «мы могли бы», «я взял ответственность за X». |

## 5. Правила вычисления ступени (composition rule)

### 5.1 Общая ступень (stage)

Для стержневых слотов (W, M1, M2, M4) каждый имеет индекс 1-5. **Ступень = min** по мастерственным слотам:

```
stage_raw = min(M1.baseline_idx, M2.baseline_idx, M4.baseline_idx)
```

Используется `baseline`, не `value-now` — программа не реагирует на шум (PD.FORM.088 §2.1).

### 5.2 Мировоззренческий gate для Д→П (FORM.080 §7.1)

> **Уточнение 4 мая 2026.** Этот gate — gate **по рычагам** (RCS-слоты W, M1-M4, A). Полный тройной gate выпускника программы ЛР включает дополнительно: (i) gate по характеристикам выпускника (PD.FORM.093 §5.4: 5 gate-характеристик ≥4 + 3 supporting ≥3), (ii) gate по реальному изменению (PD.FORM.093 §7.2: ≤3 вопроса Диагноста, confidence ≥ medium). Без обоих дополнений рычаги могут быть «поставлены», а характеристики — не проявлены, или проявлены без реального изменения в физ.мире (PD.FAIL.055).

**Gate по рычагам (этот раздел):** для перехода ст. 4 → 5:
- **Метрический gate:** `stage_raw ≥ 4` (все мастерственные слоты на baseline ст. 4)
- **Мировоззренческий gate:** `W.baseline_idx ≥ 4` (фаза «мир — система, и я в ней деятель»)
- **Agency gate:** `A.baseline_idx ≥ 4` (устойчивая доля самоинициированных действий)

Все три одновременно — необходимое, но не достаточное условие. Достаточность даёт PD.FORM.093 (см. выше).

```
# Gate по рычагам (RCS, этот раздел)
if stage_raw >= 4 and W.baseline_idx >= 4 and A.baseline_idx >= 4:
    rcs_gate_passed = True
else:
    stage = min(stage_raw, W.baseline_idx)   # мировоззрение не может отставать на 2+
    rcs_gate_passed = False

# Полный gate выпускника (FORM.080 §7.1, тройной)
graduate_ready = rcs_gate_passed AND form093_metric_gate AND form093_real_change_gate
```

### 5.3 Bottleneck

**Bottleneck** = слот, который определил min:

```
bottleneck = argmin(W, M1, M2, M4).baseline_idx
```

Портной и Навигатор используют bottleneck для фокуса контента на квартал. Пример: bottleneck = M2 → квартал идёт в укрепление IWE-среды и ОРЗ-ритуалов.

### 5.4 Вспомогательные слоты не входят в stage

M3, IT, A (кроме gate на ст. 5) — **профиль, не ступень**. Они влияют на:
- IT → канал доставки (Оркестратор)
- M3 → контент ст. 5 (Портной)
- A → gate перехода на ст. 5

## 6. Применение ролями платформы

### 6.1 Диагност (MIM.R.009)

**Алгоритм: три фазы, ≤5 вопросов суммарно.**

**Фаза 1. Первичный срез** (2-3 вопроса, цель: определить уровень и кандидата в bottleneck)

| Слот | Якорный вопрос |
|------|---------------|
| **M1** | «Сколько часов в неделю ты реально работаешь по методам — с планированием, помидорками или ОРЗ-ритуалом? Как это выглядело последние 4-6 недель?» |
| **M2** | «Есть ли у тебя экзокортекс (Pack или DS репо)? Как часто задачи дольше 15 мин проходят через явное Открытие и Закрытие?» |
| **W** | «Когда что-то идёт не так — как ты обычно это объясняешь? Что в последний раз помешало?» (слушать: язык внешних обстоятельств vs внутренние ресурсы vs роли и системы) |
| **M4** | «Возьми любую задачу, над которой сейчас работаешь. Как бы ты описал систему, в которую она встроена — кто играет какие роли, где границы?» |

Диагност задаёт 2-3 вопроса, не все 4 — выбирает те, где по первым ответам не ясен уровень.

**Фаза 2. Уточнение bottleneck** (1-2 вопроса, цель: подтвердить слабый слот)

| Кандидат bottleneck | Уточняющий вопрос |
|--------------------|--------------------|
| **M1 bottleneck** | «После длинного перерыва (болезнь, отпуск, аврал) — сколько времени обычно уходит на возврат к ритму?» |
| **M2 bottleneck** | «Сколько WP-контекстов сейчас открыто? Когда последний раз делал Week Close?» |
| **W bottleneck** | «Приходилось ли тебе недавно объяснять другому, как ты принял важное решение? Что ты говоришь — через события или через выбор ролей?» |
| **M4 bottleneck** | «Можешь ли показать артефакт (запись, разбор, WP-контекст), где ты использовал системный язык — роль, граница, интерфейс, обещание?» |

**Фаза 3. Аудит артефактов** (без вопросов — проверка доступных данных)

| Слот | Данные для аудита |
|------|-------------------|
| **M1** | WakaTime или тайм-трекер: среднее часов/нед за 4-8 нед; Activity Hub: частота сессий |
| **M2** | Git: наличие Pack/DS репо и дата последнего коммита; количество WP-контекстов; commits с тегами `week-close`, `day-close` |
| **W** | Анализ речи в диалоге: бытовые нарративы vs системный язык (роль, метод, ресурс, граница) |
| **M4** | Качество WP-контекстов: есть ли системные разборы, явные границы задач, описания ролей |
| **IT** | Какой канал пользователь использовал для связи с Диагностом (бот / web / VS Code) |
| **A** | Соотношение: инициированные vs assigned задачи за последние 4 нед (из Activity Hub или самооценка) |

**Вывод Диагноста** — не «ты на ступени 3», а структурированный профиль:

```
Ступень: 3 (Систематический)
Bottleneck: M2 (IWE/ОРЗ, idx=2)
Профиль: W=3, M1=3, M2=2, M4=3 | IT=3, A=3
Gap до ст.4: укрепить экзокортекс (M2 baseline 2→3)
```

Передаётся Навигатору как вход для квартального фокуса.

### 6.2 Навигатор (MIM.R.007)

Использует bottleneck для формулировки «что дальше»: квартальная цель = поднять baseline слота-bottleneck на +1 индекс.

### 6.3 Портной (MIM.SOP.001)

Выбор контента:
- Фокус-контент (квартал) = bottleneck-слот, ст. 1→3: методы двигают baseline
- Фокус-контент ст. 3→5: расширение потенциала через M4 (системное мышление)
- IT-slot определяет **канал** (бот vs VS Code), не содержание

### 6.4 Оркестратор (WP-203)

Видит полный RCS-профиль + каскад планирования (квартал → неделя → день, WP-245 Ф26). Триггеры корректировки плана на день:
- Резкое отклонение `value-now` от `baseline` по слоту M1 → защитная реакция (не новые нагрузки, а восстановление ритма)
- Baseline-рост слота = сигнал двигаться к следующей ступени

## 8. Ограничения текущей версии (known gaps)

1. **Калибровка границ.** Рубрики baseline-индексов добавлены для всех 7 слотов. Требуют валидации на пилоте — особенно границы idx 3↔4 и 4↔5. Статус: **ожидает v3** (пилоты Паша/Дима/Ильшат, май 2026).
2. **M3 (Проф-методы).** Рубрики — минимальная домен-независимая версия. Домен-специфичные рубрики (Knowledge Worker, UX, Engineer и др.) — в **v4**, после проектирования программы «Рабочее развитие».
3. **Композиция при неполных данных.** Если по слоту не удалось определить baseline (мало артефактов, новый пользователь), правило §5 не покрывает случай. Нужен fallback: при N < 2 артефактов по слоту → считать idx=1 с флагом `low_confidence`. Статус: **техдолг, не блокер**.
4. **A (Agency) сложно измерить автоматически.** Соотношение инициированных vs assigned задач не всегда видно в Activity Hub (особенно работа вне IWE). Self-report + диагностический диалог — единственный источник на ранних ступенях.

## 12. MVP: Автоматический расчёт ступени (короткий путь, WP-310)

> **Статус:** v3, май 2026 (WP-310 Ф1-Ф3). Добавляет автоматический расчёт по событиям Activity Hub поверх ручной диагностики §4-§6.
> **Отношение к §4-§6:** не заменяет Диагноста — дополняет как baseline-proxy для новых пользователей. При наличии живой диагностики — использовать её результат.
> **Весовая конфигурация:** `DS-ecosystem-development/…/Data-Governance/iwe-actions-catalog.md §7`.

### 12.1 Модель 5 характеристик

| Характеристика | Ось | Тип | Показатель | Что измеряет |
|---|---|---|---|---|
| **Систематичность** | Мастерство | обязательная | `self_dev_days_per_week` | Регулярность саморазвития: среднее дней/нед в скользящем окне |
| **Инвестированное время** | Мастерство | обязательная | `avg_hours_per_week` | Объём вложенного времени в практику (часов/нед) |
| **Методичность мышления** | Мастерство | компенсаторная | `methodical_events_per_month` | Применение методов, рост знания (событий/мес) |
| **Системность мировоззрения** | Мировоззрение | основная | `worldview_score` | Рефлексия, системный язык, стратегическое мышление |
| **Агентность** | Мировоззрение | основная | `agency_score` | Самоинициированные действия, инициативы |

> **Различение типов.** «Обязательная» = ноль блокирует ступень даже при высоких других. «Компенсаторная» = низкое значение не блокирует, если остальные компенсируют. «Основная» = наибольший вес в оси Мировоззрения.
>
> **Именование vs FORM.093.** Характеристика «Систематичность» ≠ «Собранность» из FORM.093 (там composure_idx). Это разные уровни: FORM.089 §12 — рычаговый прокси для авто-расчёта; FORM.093 — выходная характеристика выпускника. Не смешивать.

### 12.2 Показатели и рубрики

#### Систематичность — `self_dev_days_per_week`

**Смысл:** регулярность именно саморазвития, а не любой IWE-активности. `git_commit`, `wp_created`, `command_invoked` — это рабочая активность, она не считается. Считаются только дни, где человек целенаправленно учится или рефлексирует.

**Принцип окна:** окно расчёта растёт вместе со ступенью (`i.accounting_period`). Начинающий оценивается за 1 неделю, Проактивный — за 24 недели. Это гарантирует, что высокая ступень требует доказанной устойчивости на длинной дистанции.

**Показатель:** `COUNT(DISTINCT active_days) / accounting_weeks` за окно текущей ступени.

**Множество событий `SELF_DEV_EVENT_TYPES`:**

```python
SELF_DEV_EVENT_TYPES = {
    # Обучение
    "lesson_completed",
    "knowledge_extracted",
    "pack_updated",
    "qualification_granted",
    # ОРЗ-ритуалы (практика саморазвития)
    "day_plan_opened", "day_open",        # day_open = legacy alias (iwe.py)
    "day_plan_closed", "day_close",       # day_close = legacy alias (ORZ hook до переименования)
    "week_plan_closed",
    "month_plan_closed",
    "strategy_session_completed",
    # IWE-среда
    "iwe_session",
}
```

**НЕ входят:** `git_commit`, `wp_created`, `wp_closed`, `wp_completed`, `command_invoked`, `slot_logged`, `coding_time`, `club_post_created` — рабочая, инструментальная или техническая активность.

> **Gap-Д (архитектурный):** текущий M1 в `recalculate_derived.py` использует `activity_domain IN ('practice', 'learning')` — шире SELF_DEV_EVENT_TYPES (включает git_commit, wp_created и т.д.). Для Аттестатора нужен отдельный запрос только по SELF_DEV_EVENT_TYPES. Фикс в rcs-collector — следующая фаза WP-310.
>
> **Gap-Б (minor):** ORZ hook emits `day_close` (legacy), не `day_plan_closed`. iwe.py emits `day_open`. Оба alias в SELF_DEV_EVENT_TYPES до миграции.

| **2** | Практикующий | ≥ 3 | 4 нед | ≥ 3 дней/нед в среднем за 4 недели |
| **3** | Систематический | ≥ 5 | 8 нед | ≥ 5 дней/нед в среднем за 8 недель |
| **4** | Дисциплинированный | ≥ 6 | 12 нед | ≥ 6 дней/нед в среднем за 12 недель |
| **5** | Проактивный | ≥ 6.7 | 24 нед | ≥ 6.7 дней/нед в среднем за 24 недели |

#### Инвестированное время — `avg_hours_per_week`

**Принцип окна:** то же, что у Систематичности — окно растёт со ступенью (`i.accounting_period`). Среднее считается за период, соответствующий текущей ступени.

**Источники (суммируются):**
- `coding_time` — `payload.total_seconds / 3600` (WakaTime через iwe.py adapter)
- `slot_logged` — `payload.hours`
- `lesson_completed` — `payload.duration_minutes / 60`

| Индекс | Ступень-аналог | Порог ч/нед | Окно | Операционные признаки |
|--------|:---:|:---:|:---:|---|
| **0** | — | < 0.5 | — | Практически нет вложений (< 30 мин/нед) |
| **1** | Случайный | ≥ 0.5 | 1 нед | Есть что-то. Разовая сессия за неделю |
| **2** | Практикующий | ≥ 4 | 4 нед | ≥ 4 ч/нед в среднем за 4 недели |
| **3** | Систематический | ≥ 6 | 8 нед | ≥ 6 ч/нед в среднем за 8 недель |
| **4** | Дисциплинированный | ≥ 8 | 12 нед | ≥ 8 ч/нед в среднем за 12 недель |
| **5** | Проактивный | ≥ 10 | 24 нед | ≥ 10 ч/нед в среднем за 24 недели |

#### Методичность мышления — `methodical_events_per_month`

**Показатель:** `COUNT(events за 30 дней)` из типов: `lesson_completed`, `knowledge_extracted`, `pack_updated`, `qualification_granted`.

| Индекс | Диапазон | Операционные признаки |
|--------|----------|----------------------|
| **0** | 0 событий | Методы не применяются |
| **1** | 1–3/мес | Редкое применение, скорее случайное |
| **2** | 4–10/мес | Пробует методы в задачах. Начало формирования привычки |
| **3** | 11–25/мес | Методы — часть регулярной практики. Pack растёт |
| **4** | 26–50/мес | Системная методическая работа. Pack растёт еженедельно |
| **5** | > 50/мес | Методы — ядро практики. Извлечение и формализация ежедневно |

#### Системность мировоззрения — `worldview_score`

**Показатель:** взвешенная сумма событий за 30 дней, нормированная в индекс 0–5. Веса: `iwe-actions-catalog.md §7`.

| Индекс | Операционные признаки |
|--------|----------------------|
| **0** | Рефлексий нет. Язык бытовой («не получилось», «повезло») без системного |
| **1** | Единичные рефлексии. Видит связи в знакомых ситуациях |
| **2** | Начинает видеть паттерны. Системный язык — эпизодически |
| **3** | Регулярные рефлексии (Week Close / Month Close). Различает роли, методы, системы |
| **4** | Чёткий системный язык. Видит себя как деятеля в системах |
| **5** | Формирует системное видение у других. Экспортирует мировоззрение |

События-источники (с весами в каталоге): `week_plan_closed` (q≥3), `month_plan_closed`, `strategy_session_completed`, `knowledge_extracted`, `pack_updated`.

#### Агентность — `agency_score`

**Показатель:** взвешенная сумма событий за 30 дней, нормированная в индекс 0–5. Веса: `iwe-actions-catalog.md §7`.

| Индекс | Операционные признаки |
|--------|----------------------|
| **0** | Нет инициированных действий. Действует только по запросу |
| **1** | 1–2 самоинициированных действия/мес. В основном реактивный |
| **2** | Несколько инициированных задач. Есть хотя бы 1 самостоятельный РП |
| **3** | ~50% задач из собственных целей. Держит 1–2 активных РП |
| **4** | Несколько независимых РП. Действует в неопределённости |
| **5** | Формирует повестку для ближнего круга. Инициирует изменения в системах |

События-источники: `wp_created` (MVP: все считаются самоинициированными — нет поля `initiator`, ждём WP-214 Ф10.5), `wp_closed`, `wp_completed`, `strategy_session_completed`.

#### Устойчивость слотов — `execution_stability` (i.execution_stability)

Дополняет Систематичность: проверяет не только среднюю частоту дней/нед, но и серийность (consecutive streaks). Серии — признак автоматизированного ритма, а не случайных всплесков.

| Индекс | Ступень-аналог | Критерий | Окно |
|--------|:---:|---|:---:|
| **0** | — | Нет серий | 1 нед |
| **1** | Случайный | Нет требования | 1 нед |
| **2** | Практикующий | ≥ 1 серия из ≥ 3 дней подряд | 4 нед |
| **3** | Систематический | Серии из ≥ 4 дней регулярно | 8 нед |
| **4** | Дисциплинированный | Серии из ≥ 5 дней, разрывы редки | 12 нед |
| **5** | Проактивный | Серии из ≥ 6 дней, разрывы — единицы | 24 нед |

> **MVP:** execution_stability логируется, но в MVP не блокирует ступень (нет реализации в rcs-collector). Используется как сигнал при равных значениях Систематичности. Добавить в коллектор — Ф4+ WP-310.

#### Минимальные накопленные часы — hard gate квалификации (i.min_hours_for_qualification)

Жёсткий кумулятивный барьер: нельзя получить ступень, не накопив минимальный суммарный опыт, независимо от текущей частоты.

| Ступень | Минимум накопленных часов |
|---|:---:|
| Практикующий (2) | 20 ч |
| Систематический (3) | 48 ч |
| Дисциплинированный (4) | 96 ч |
| Проактивный (5) | 161 ч |

Источник: суммарно из `slot_logged.hours` + `lesson_completed.duration_minutes/60` + `coding_time.total_seconds/3600` за всё время аккаунта.

### 12.3 Матрица минимумов ступеней

| Характеристика | Ст. 1 | Ст. 2 | Ст. 3 | Ст. 4 | Ст. 5 (rcs-часть) |
|---|:---:|:---:|:---:|:---:|:---:|
| **Систематичность** (обяз.) | — | ≥ 2 | ≥ 3 | ≥ 4 | ≥ 5 |
| **Инвестированное время** (обяз.) | — | ≥ 2 | ≥ 3 | ≥ 4 | ≥ 5 |
| **Накоп. часы** (hard gate) | — | ≥ 20 ч | ≥ 48 ч | ≥ 96 ч | ≥ 161 ч |
| **Методичность мышления** | — | — | ≥ 1 | ≥ 2 | ≥ 3 |
| **Системность мировоззрения** | — | — | ≥ 2 | ≥ 3 | ≥ 4 |
| **Агентность** | — | — | ≥ 1 | ≥ 2 | ≥ 4 |

Ступень 1 = новый пользователь, нет требований. «—» = нет минимума. Индексы 2-5 соответствуют Практикующий-Проактивный порогам из §12.2 рубрик.

**Период учёта по ступени (`i.accounting_period`):**

| Ступень | Окно расчёта |
|---|---|
| Случайный (1) | 1 нед |
| Практикующий (2) | 4 нед |
| Систематический (3) | 8 нед |
| Дисциплинированный (4) | 12 нед |
| Проактивный (5) | 24 нед |

SQL для Систематичности использует `NOW() - INTERVAL '{accounting_weeks * 7} days'` на основе предыдущей ступени пользователя.

**Алгоритм расчёта:**

```python
ACCOUNTING_WEEKS = {1: 1, 2: 4, 3: 8, 4: 12, 5: 24}
CUMULATIVE_HOURS_GATE = {2: 20, 3: 48, 4: 96, 5: 161}

def compute_stage_mvp(s: int, t: int, m: int, w: int, a: int,
                      total_hours: float = 0) -> int:
    """
    s = систематичность, t = инвестированное_время,
    m = методичность, w = системность, a = агентность — индексы 0–5.
    total_hours = накопленные часы всего (hard gate квалификации).

    Индексы прямо соответствуют ступени-аналогу:
      idx 2 = Практикующий порог, idx 3 = Систематический, и т.д.
    """
    # mandatory gate: оба обязательных = 0 → ступень 1
    if s == 0 or t == 0:
        return 1
    # характеристиковый gate
    if s >= 5 and t >= 5 and m >= 3 and w >= 4 and a >= 4:
        stage = 5  # rcs_gate_passed — ещё нужен FORM.093 gate
    elif s >= 4 and t >= 4 and m >= 2 and w >= 3 and a >= 2:
        stage = 4
    elif s >= 3 and t >= 3 and m >= 1 and w >= 2 and a >= 1:
        stage = 3
    elif s >= 2 and t >= 2:
        stage = 2
    else:
        stage = 1
    # hard gate: накопленные часы — понижаем до тех пор, пока gate пройден
    while stage >= 2 and total_hours < CUMULATIVE_HOURS_GATE.get(stage, 0):
        stage -= 1
    return stage
```

> Ступень 5 из этой функции — `rcs_gate_passed`. Итоговая ступень 5 требует дополнительно `form093_metric_gate AND form093_real_change_gate` (FORM.093 §5.4, §7.2).

### 12.6 Известные ограничения MVP

1. **Агентность без initiator.** Поле `initiator` в `wp_created` отсутствует (WP-214 Ф10.5). MVP: все wp_created = самоинициированные. После WP-214: только `initiator=user`.
2. **WakaTime опциональный.** Если не подключён — инвестированное время может быть занижено. Fallback: только `slot_logged + lesson_completed`.
3. **Качество Week Close.** Поле `quality` в `week_plan_closed` payload используется для весового бонуса Системности мировоззрения — требует корректного заполнения хуком (Gap-А).
4. **Discourse-клуб не подключён.** `club_post_created` (WP-296, ждёт ORY-SSO) добавит вес в Системность мировоззрения при подключении.
5. **Gap-Д: rcs-collector использует широкий фильтр.** Текущий M1 в `recalculate_derived.py` считает `activity_domain IN ('practice', 'learning')` — включает `git_commit`, `wp_created` и т.д. Аттестатор требует отдельного запроса по SELF_DEV_EVENT_TYPES. Реализация — Ф4+ WP-310.
6. **Устойчивость слотов не реализована в коллекторе.** `execution_stability` (серии дней подряд) пока только в спецификации; нет в rcs-collector. Ф4+ WP-310.
7. **Накопленные часы — приближение.** Кумулятивный счёт часов из Activity Hub может занижать реальные инвестиции (если WakaTime не подключён с первого дня). На первых волнах — допустимо.
