# Формализация: Role Characterisation Space (RCS) Ученика

## 1. Назначение

### Четыре понятия — четыре разных вопроса

RCS использует четыре термина, которые отвечают на разные вопросы про Созидателя. Без этого различения они кажутся взаимозаменяемыми — но это не так.

| Вопрос | Термин | Пример |
|--------|--------|--------|
| **Из чего состоит Созидатель?** | **Узел** | Организм, Репозитории, Агенты |
| **В каком направлении он растёт?** | **Ось** | Мировоззрение / Мастерство (Пилот); БД / Программа (Машина) |
| **Что счита́ет машина автоматически из событий?** | **bh-характеристика** (bh. = behaviour) | bh.sys, bh.inv, bh.met — 7 метрик Аттестатора |
| **Как Диагност увидит потенциал в диалоге?** | **cp-срез** (cp. = capacity) | cp.rhy, cp.wld, cp.int — 13 срезов Диагноста |

Узел — физический объект. Ось — направление роста. bh-характеристика — непрерывный счёт поведения из событий. cp-срез — точечная оценка потенциала через рубрику. Четыре понятия, потому что четыре разных вопроса.

### Болид-онтология

Созидатель = **Болид**: Пилот (Человек) + Машина (IWE). Три объекта развиваются **независимо**:

| Объект | Как развивается | Кто развивает |
|--------|-----------------|---------------|
| **Пилот** | Тренируется — наращивает Мировоззрение и Мастерство | Человек через программу ЛР |
| **Машина** | Строится — пополняются Pack, Среда, Агенты | Человек при проектировании IWE |
| **Болид** | Обкатывается — пилот и машина притираются в работе | Совместная практика реальных проектов |

**Создание Созидателя** (болид-уровень) = создание IWE + развитие Пилота + тренировка совместимости.

### Узлы Пилота

```
Человек (Пилот):
  Организм                          — тело, ритм, энергия
  Личность:
    Мыслительное мастерство         — картина мира, системные различения
    Прикладное мастерство:
      Мастерство саморазвития       — роль Ученика, M001–M008 ← ЛР
      Мастерство управления IWE     — ОРЗ-ритуалы, экзокортекс, агенты ← ЛР
      Мастерство создания IWE       — Pack-дизайн, новые методы, MCP-инструменты (ст.3+)
      Профессиональные методы       — методы предметной области (вне ЛР)
      Прочие прикладные мастерства  — спорт, язык, ремесло и т.д. (вне ЛР)
```

> **ЛР работает на три узла:** Мыслительное мастерство, Мастерство саморазвития, Мастерство управления IWE. Создание IWE начинается на ст.3+ и развивается в программе «Рабочее развитие».

### Узлы Машины

```
IWE (Машина):
  Репозитории (БД)                  — Pack и DS-файлы: формализованные знания
  Среда (Программа)                 — VS Code, Git, хуки, конфигурации, протоколы
  Агенты                            — модели + настройки + MCP-инструменты
```

### Оси роста

| Объект | Воздействие → Ось | Что меняется |
|--------|-------------------|--------------|
| **Пилот** | теория → **Мировоззрение** | Как видит себя, мир, роли |
| **Пилот** | методы → **Мастерство** | Что умеет делать устойчиво |
| **Машина** | информация → **БД** | Накопленные формализованные знания |
| **Машина** | инструменты → **Программа** | Среда, скрипты, агентные навыки |

### Петля управления (определение ступени)

Ступень не определяется ни только Аттестатором, ни только Диагностом — нужен **двойной gate**:

```
bh-характеристики (Аттестатор, авто) → сигнал → Диагност (cp-срезы, по запросу) → подтверждение → Ступень
```

1. Аттестатор считает 7 bh-метрик из событий Activity Hub → сигнал «рекомендуемая ступень»
2. Диагност проводит диалог по ≤5 вопросам → оценивает потенциал по cp-срезам → подтверждает или корректирует
3. Итоговая ступень фиксируется только после подтверждения Диагноста

**Зачем двойной gate?** bh-метрики фиксируют частоту и объём поведения (что и сколько делает), но не видят потенциал (глубину понимания, качество мышления, возможность роста). cp-срезы видят потенциал, но не частоту. Только вместе они дают воспроизводимую ступень.

### Два инструмента измерения

| Инструмент | Коды | Когда | Метод | Что измеряет |
|------------|------|-------|-------|--------------|
| **Аттестатор** (WP-310) | **bh.*** | Непрерывно, каждую неделю | SQL по Activity Hub, §12 | Поведение: частота, объём, регулярность |
| **Диагност** (MIM.R.009) | **cp.*** | По запросу, при расчёте ступени | ≤5 вопросов + рубрики §4, аудит артефактов | Потенциал: глубина понимания, качество ОРЗ |

### Проблема и решение

**Проблема.** PD.FORM.003 и PD.FORM.080 описывают ступени Ученика (1 Случайный → 5 Проактивный) как плоские нарративные состояния. При диагностике живого человека решение «ступень = N» не воспроизводимо — два диагноста могут прийти к разным ступеням.

**Решение.** Перейти к **многомерному профилю** по формализму FPF A.19 `U.CharacteristicSpace`. Разделить автоматические bh-метрики (Аттестатор) и диагностические cp-срезы (Диагност). Ступень вычисляется из профиля явным правилом (§5).

**Три эффекта:**
1. **Диагностика воспроизводима.** Алгоритм: bh-сигнал → cp-рубрики → ступень + bottleneck.
2. **Видно слабое звено.** Bottleneck определяет фокус на квартал.
3. **Двойной gate Д→П** (FORM.080 §7.1) формализован: bh.sys/inv/met/awr/agn + cp-подтверждение.

## 2. Структура RCS: 13 cp-срезов (диагностика Диагноста)

> **Принцип.** Каждый cp-срез диагностирует конкретный узел Созидателя через рубрику 1-5. Срез ≠ узел: узел = физический объект, срез = инструмент измерения потенциала.
>
> **TTL правило.** Cp-срезы категории **mandatory** (блокирующие) имеют срок действия **6 месяцев**. Диагноз старше 6 месяцев — требует повторной диагностики перед переходом на следующую ступень.

| Код | Срез | Что диагностирует | Узел | Статус |
|-----|------|-------------------|------|--------|
| **cp.rhy** | Ритмичность | Регулярность практики: частота, ритм, восстановление после срыва | Организм | **mandatory** |
| **cp.wld** | Мировоззрение | Зрелость системного взгляда (мировоззренческая дуга FORM.080 §3) | Мыслительное мастерство | **mandatory** |
| **cp.skl** | Мастерство саморазвития | Владение M001–M008: глубина и автоматизм применения | Мастерство саморазвития | **mandatory** |
| **cp.iwe** | Управление IWE | ОРЗ-ритуалы, Pack/DS-структуры, маршрутизация знаний, Week/Month Close | Мастерство управления IWE | **mandatory** |
| **cp.int** | Системное мышление | Применение различений: роль, система, граница, эмерджентность | Мыслительное мастерство | **mandatory** |
| **cp.agt** | Агентность | Доля самоинициированных действий, независимость решений | Поведенческий gate (§5.2) | **mandatory** (gate ст.5) |
| **cp.cre** | Создание IWE | Pack-дизайн, MCP-инструменты, расширения протоколов | Мастерство создания IWE | informational |
| **cp.knw** | Системное знание | Владение SoTA-методами домена | Профессиональные методы | informational |
| **cp.env** | Среда и окружение | Осознанное управление физической и цифровой средой | Среда (IWE.Среда) | informational |
| **cp.auto** | Автопилот | Глубина использования AI-агентов как автопилота | Агенты (IWE.Агенты) | informational |
| **cp.exo** | Экзоскелет | Использование IWE как усилителя мышления, не замены | Среда + Агенты | informational |
| **cp.cult** | Культура и передача | Передача методов и мировоззрения другим | Надсистема (сообщество) | informational |
| **cp.traj** | Траектория и рефлексия | Осознанное управление собственным путём развития | Личность (рефлексия) | informational |

**Стержневые для расчёта ступени:** cp.rhy, cp.wld, cp.skl, cp.iwe, cp.int + cp.agt (gate ст.5).
**Информационные (профиль, не блокируют):** cp.cre, cp.knw, cp.env, cp.auto, cp.exo, cp.cult, cp.traj.

## 3. Квартет каждого среза (по PD.FORM.088 §2)

Каждый cp-срез = четвёрка:

| Элемент | Что фиксируется |
|---------|-----------------|
| **Характеристика** | Свойство системы-человека как аспект внимания |
| **Показатель** | Выбранная шкала и единица измерения |
| **Значение** | Два числа: **сейчас** (моментальное) и **baseline** (устойчивый уровень, сглаживание 4-8 недель) |
| **Потенциал** | Текущий потолок шкалы для данного человека |

**Правило формулировки цели:** любая цель программы ЛР в рамках среза обязана называть, что двигаем:
- «Поднять **baseline** среза cp.rhy с N до M за квартал» → cp.skl/cp.iwe (методы)
- «Расширить **потенциал** среза cp.int с N до M за год» → cp.wld/cp.int (фундаментальное развитие)

Цель, сформулированная через «значение-сейчас», отбрасывается как реакция на шум (PD.FORM.088 §2.1).

## 4. Определения cp-срезов

> **Ограничение MVP:** конкретные шкалы и пороги — draft. Финальная версия со всеми тестовыми заданиями и алгоритмом измерения — следующая итерация (§9 Roadmap).

### cp.rhy — Ритмичность

| Элемент | Значение |
|---------|----------|
| Характеристика | Регулярность практики саморазвития: ритм, частота, восстановление |
| Показатель | Среднее дней/нед практики (скользящее 4-8 нед) + max разрыв |
| Значение (сейчас) | Дней/нед текущей недели |
| Значение (baseline) | Скользящее среднее 4-8 недель |
| Потенциал | Ежедневная практика как стиль жизни, max разрыв ≤1 дня |

**Рубрики baseline-индексов 1-5.**

| Индекс | Дн/нед | Операционные признаки baseline |
|--------|:------:|--------------------------------|
| **1** | ~1-2 | Случайные всплески. Нет паттерна недели. Срывы на месяцы. Восстановление ≥ месяца |
| **2** | ~3 | Пробует регулярность. Минимум 1-2 фиксированных слота в неделю. Срывы 2-3 нед, восстановление 2-3 нед |
| **3** | ~5 | Устойчивый недельный ритм. Срывы короткие (≤2 нед), восстановление ≤1 нед. Практика — привычка, не усилие воли |
| **4** | ~6 | Высокая регулярность. Редкие срывы (≤7 дней). Ритм адаптируется к контексту (командировки, аврал) — не разрушается |
| **5** | ~6.7+ | Ежедневная практика как стиль. Пропуск ≤1-2 дня/мес. Ритм передаётся — способен поддержать другого в построении ритма |

### cp.wld — Мировоззрение

| Элемент | Значение |
|---------|----------|
| Характеристика | Зрелость системного мировоззрения |
| Показатель | Фаза мировоззренческой дуги (FORM.080 §3), индекс 1-5 |
| Значение (сейчас) | Текущая фаза по ответам в диалоге |
| Значение (baseline) | Фаза, устойчиво проявляющаяся 4+ нед в артефактах |
| Потенциал | 5 (системное мировоззрение + agency) |

**Рубрики baseline-индексов 1-5.**

| Индекс | Операционные признаки baseline |
|--------|--------------------------------|
| **1** | Проблемы объясняет внешними обстоятельствами. Нет языка системы/метода. Артефакты: отсутствуют или чисто событийные |
| **2** | Видит внутренние ресурсы: внимание, энергия, ритм. Различает «состояние» и «задачу». В речи: «я собран/не собран», «у меня не было ресурса» |
| **3** | Замечает влияние среды (люди, мемы, пространство). В речи: «окружение тянет», «я попал под влияние». Артефакты: записи о влиянии среды |
| **4** | Описывает задачи через роли, границы, интерфейсы. Различает «я» и «моя роль». В речи: «я как [роль]», «обещание системы». Артефакты: системные разборы ситуаций |
| **5** | Интегрированное системное мировоззрение + agency. Создаёт культурные формы. В речи: «мы как система». Артефакты: наставничество, публикации, инициативы |

### cp.skl — Мастерство саморазвития

| Элемент | Значение |
|---------|----------|
| Характеристика | Глубина и автоматизм применения методов M001–M008 |
| Показатель | Уровень освоения методов: знает / применяет эпизодически / применяет системно / учит других |
| Значение (сейчас) | Применяемые методы текущей недели |
| Значение (baseline) | Устойчиво встроенные методы (6+ нед применения) |
| Потенциал | Все методы ЛР как автоматизированный стиль + способность передать |

**Рубрики baseline-индексов 1-5.**

| Индекс | Операционные признаки baseline |
|--------|--------------------------------|
| **1** | Методы ЛР (M001-M008) неизвестны или известны формально. Работа по интуиции и привычке. Нет учёта времени, слотов или Day Open/Close |
| **2** | Знает 2-3 метода ЛР. Применяет помидорки или Day Open/Close эпизодически. Держит ритм 2-3 нед, потом срыв. Восстановление — несколько недель |
| **3** | 4-6 методов ЛР применяет устойчиво. Помидорки/слоты как привычка. Day Open/Close регулярно. Восстановление после срыва — ≤1 нед |
| **4** | Полный стек M001-M008 применяется автоматически. Методы встроены в ежедневный режим без напоминаний. Понимает граничные случаи применения |
| **5** | Методы ЛР — стиль жизни. Адаптирует методы под свой контекст. Способен передать метод другому и поддержать его освоение |

### cp.iwe — Управление IWE

| Элемент | Значение |
|---------|----------|
| Характеристика | Владение рабочей средой: ОРЗ-ритуалы, Pack/DS-структуры, маршрутизация знаний, управление памятью, конфигурация агентов |
| Показатель | Зрелость экзокортекса: доля задач через ОРЗ + наличие Pack/DS + активность Week/Month Close |
| Значение (сейчас) | % задач за неделю через ОРЗ |
| Значение (baseline) | Устойчивая доля + зрелые артефакты (WP-контексты, captures, Week/Month Close) |
| Потенциал | 100% значимых задач через ОРЗ + полная IWE-инфраструктура |

**Рубрики baseline-индексов 1-5.**

| Индекс | Операционные признаки baseline |
|--------|--------------------------------|
| **1** | ОРЗ не применяется. Задачи без явного Открытия/Закрытия. Экзокортекса нет (возможно, есть бот как канал). Артефакты случайны и не связаны. Нет WP-контекстов |
| **2** | Open/Close применяются эпизодически (<30% значимых задач). Начата Pack- или DS-структура, слабо заполнена. 1-2 WP-контекста. Day Open/Close — нерегулярно. Week Close — нет |
| **3** | ОРЗ на значимых задачах (>60%). Day Open/Close — регулярно. Pack и DS структурированы по шаблонам. 5+ WP-контекстов. Week Close — регулярно. Memory-файлы обновляются |
| **4** | ОРЗ как норма (>90%). Протоколы «на автомате». Зрелый экзокортекс: Pack с 10+ формализациями, DS с чётким routing. Month Close как ритуал с явным циклом обратной связи. Субагенты применяются |
| **5** | 100% значимых задач через ОРЗ. Свои расширения протоколов, задокументированные для других. Передаёт метод. Экзокортекс — полная инфраструктура: Pack, DS, FMT-шаблоны, staging-канал |

### cp.int — Системное мышление

| Элемент | Значение |
|---------|----------|
| Характеристика | Способность к системному взгляду и системному разбору |
| Показатель | Применение различений (роль, система, граница, эмерджентность, деятельность) в разборе реальных ситуаций |
| Значение (сейчас) | Сколько различений применил в недельных задачах |
| Значение (baseline) | Стабильно применяемые различения |
| Потенциал | Интегративная сложность + калибр задач, которые способен удержать |

**Рубрики baseline-индексов 1-5.** Измеряется разбором реальных ситуаций (не тест-викториной).

| Индекс | Операционные признаки baseline |
|--------|--------------------------------|
| **1** | Различения не применяются. Язык бытовой, проблемы объясняются нарративами. Путает систему с группой людей, роль с должностью |
| **2** | Применяет 1-2 базовых различения (система vs процесс, метод vs инструмент). Частые ошибки. В незнакомой области — беспомощен |
| **3** | Применяет 5-7 различений стабильно в знакомых контекстах. Находит роли, границы, интерфейсы. В новом контексте — нужен образец |
| **4** | Применяет 10+ различений в новых ситуациях без образца. Самостоятельно находит неочевидные системы. Конструктивно критикует чужие разборы |
| **5** | Интегрированное системное мышление. Создаёт новые различения. Удерживает калибр больших систем. Передаёт метод — обучает системному разбору |

### cp.agt — Агентность

| Элемент | Значение |
|---------|----------|
| Характеристика | Способность инициировать изменения, действовать автономно |
| Показатель | Доля самоинициированных действий vs реактивных (задания от окружения) |
| Значение (сейчас) | % за неделю |
| Значение (baseline) | Устойчивая доля + наличие собственных проектов (не заимствованных) |
| Потенциал | ≥80% самоинициированных (Проактивный, FORM.087) |

**Gate для ступени 5.** Без cp.agt ≥ 4 человек остаётся на ступени 4, даже если все mandatory cp-срезы на уровне.

**Рубрики baseline-индексов 1-5.**

| Индекс | Операционные признаки baseline |
|--------|--------------------------------|
| **1** | Действует почти исключительно по запросу. Собственных проектов нет. Решения откладываются — ждёт разрешения или чужой инициативы. В речи: «мне сказали», «когда можно будет» |
| **2** | Есть 1-2 личных намерения. Инициирует мелкие изменения. Большинство задач — от окружения. Решения принимает медленно, часто запрашивает одобрение |
| **3** | ~50% задач из собственных целей. Держит один активный личный проект. Принимает решения в знакомых контекстах без подтверждения. В незнакомом — нужен образец |
| **4** | >60% самоинициировано. Несколько независимых проектов. Действует в неопределённости: ставит эксперимент, оценивает, корректирует |
| **5** | >80% действий собственные. Формирует повестку для ближнего круга. Инициирует изменения в системах без формальной власти. Создаёт условия, в которых другие становятся проактивными |

## 5. Правила вычисления ступени (composition rule)

### 5.1 Двойной gate (Аттестатор → Диагност)

```
1. Аттестатор рассчитывает bh-сигнал по §12.3 → recommended_stage
2. Диагност проводит диагностику: ≤5 вопросов по cp-срезам → cp-профиль
3. Итоговая ступень = min(recommended_stage, cp_confirmed_stage)
```

**Порядок приоритета:** bh-сигнал не может быть выше cp-подтверждения. Человек с высокой систематичностью (bh.sys=5), но слабым мировоззрением (cp.wld=2) → ступень 2.

### 5.2 Мировоззренческий gate для Д→П (FORM.080 §7.1)

> **Уточнение.** Полный тройной gate выпускника программы ЛР включает: (i) gate по bh-характеристикам (§12.3), (ii) gate по cp-срезам (этот раздел), (iii) gate по реальному изменению (PD.FORM.093 §7.2: ≤3 вопроса Диагноста, confidence ≥ medium).

**Gate по cp-срезам (этот раздел):** для перехода ст. 4 → 5:
- **Стержневой gate:** cp.rhy ≥ 4, cp.wld ≥ 4, cp.skl ≥ 4, cp.iwe ≥ 4, cp.int ≥ 4
- **Agency gate:** cp.agt ≥ 4 (устойчивая доля самоинициированных действий)
- **TTL gate:** все blocking cp-срезы диагностированы ≤6 месяцев назад

```python
if all(cp[s] >= 4 for s in ['rhy','wld','skl','iwe','int']) and cp['agt'] >= 4:
    cp_gate_passed = True
else:
    # мировоззрение не может отставать на 2+ от мастерства
    stage = min(stage_raw, cp['wld'])
    cp_gate_passed = False
```

### 5.3 Bottleneck

**Bottleneck** = cp-срез с минимальным индексом среди mandatory:

```
bottleneck = argmin(cp.rhy, cp.wld, cp.skl, cp.iwe, cp.int).baseline_idx
```

Навигатор и Портной используют bottleneck для фокуса контента на квартал.

### 5.4 Информационные cp-срезы не блокируют ступень

cp.cre, cp.knw, cp.env, cp.auto, cp.exo, cp.cult, cp.traj — **профиль, не ступень**. Они влияют на:
- Рекомендации Навигатора
- Контент Портного
- Выбор канала (cp.auto → глубина автопилота)

## 6. Применение ролями платформы

### 6.1 Диагност (MIM.R.009)

**Алгоритм: три фазы, ≤5 вопросов суммарно.**

**Фаза 1. Первичный срез** (2-3 вопроса, цель: определить уровень и кандидата в bottleneck)

| cp-срез | Якорный вопрос |
|---------|---------------|
| **cp.rhy** | «Как выглядела твоя практика последние 4-6 недель — сколько дней в неделю, были ли срывы, как восстанавливался?» |
| **cp.iwe** | «Есть ли у тебя экзокортекс (Pack или DS репо)? Как часто задачи дольше 15 мин проходят через явное Открытие и Закрытие?» |
| **cp.wld** | «Когда что-то идёт не так — как ты обычно это объясняешь? Что в последний раз помешало?» |
| **cp.int** | «Возьми любую задачу, над которой сейчас работаешь. Как бы ты описал систему, в которую она встроена?» |

**Фаза 2. Уточнение bottleneck** (1-2 вопроса)

| Кандидат | Уточняющий вопрос |
|---------|-------------------|
| **cp.rhy bottleneck** | «После длинного перерыва (болезнь, аврал) — сколько времени уходит на возврат к ритму?» |
| **cp.iwe bottleneck** | «Сколько WP-контекстов сейчас открыто? Когда последний раз делал Week Close?» |
| **cp.wld bottleneck** | «Приходилось ли объяснять другому важное решение? Через события или через выбор ролей?» |
| **cp.int bottleneck** | «Можешь показать артефакт, где ты использовал системный язык — роль, граница, интерфейс?» |

**Фаза 3. Аудит артефактов** (без вопросов)

| cp-срез | Данные для аудита |
|---------|-------------------|
| **cp.rhy** | Activity Hub: частота SELF_DEV событий, max gap за 8 нед |
| **cp.iwe** | Git: наличие Pack/DS репо, WP-контекстов, commits с week-close/day-close |
| **cp.wld** | Анализ речи в диалоге: бытовые нарративы vs системный язык |
| **cp.int** | Качество WP-контекстов: системные разборы, явные границы, описания ролей |
| **cp.auto** | Какой канал использовал для связи с Диагностом; использует ли MCP-инструменты |
| **cp.agt** | Соотношение инициированных vs assigned задач за 4 нед (из Activity Hub или самооценка) |

**Вывод Диагноста:**

```
Ступень: 3 (Систематический)
Bottleneck: cp.iwe (idx=2)
Профиль: cp.rhy=3, cp.wld=3, cp.skl=3, cp.iwe=2, cp.int=3
Инфо: cp.agt=3, cp.exo=2, cp.auto=2
Gap до ст.4: укрепить cp.iwe (IWE-среда, baseline 2→3)
```

### 6.2 Навигатор (MIM.R.007)

Использует bottleneck для квартального фокуса: цель = поднять baseline cp-bottleneck на +1 индекс.

### 6.3 Портной (MIM.SOP.001)

- Фокус-контент (квартал) = bottleneck cp-срез
- cp.auto определяет канал доставки (глубина автопилота → тип контента)
- cp.int: ст. 3→5 расширение потенциала через M4 (системное мышление)

### 6.4 Оркестратор (WP-203)

Видит полный cp-профиль + bh-метрики (§12). Триггеры корректировки дня:
- Резкое отклонение bh.sys от нормы → защитная реакция (восстановление ритма)
- bh.stb сигнал (max gap) → предложение «мини-сессии» для возобновления

## 8. Ограничения текущей версии (known gaps)

1. **Калибровка cp-рубрик.** Рубрики baseline-индексов требуют валидации на пилоте — особенно границы idx 3↔4 и 4↔5. Статус: **ожидает v5** (пилоты, лето 2026).
2. **cp.knw (Системное знание).** Рубрики — минимальная домен-независимая версия. Домен-специфичные рубрики (KW, UX, Engineer) — в **v5**, после проектирования программы «Рабочее развитие».
3. **cp.agt измерение.** Поле `initiator` в `wp_created` отсутствует (WP-214 Ф10.5). MVP: все wp_created = самоинициированные. После WP-214: только `initiator=user`.
4. **bh.scl без данных actual_hours.** Поле реального времени по wp_completed появится после доработки Activity Hub. До тех пор bh.scl использует фиксированный вес из iwe-actions-catalog.md.
5. **TTL cp-срезов не реализован автоматически.** Диагност обязан вручную проверять дату последнего диагноза. Автоматический TTL-триггер — WP-310 Ф5+.
6. **cp.auto / cp.exo сложно разделить.** В реальном диагностическом диалоге граница автопилота и экзоскелета размыта. MVP: вопрос «можешь объяснить без "ИИ подсказал"?» как разделитель.

## 12. MVP: Автоматический расчёт ступени (bh-характеристики, WP-310)

> **Статус:** v4, май 2026 (WP-310 Ф1-Ф9). 7 bh-характеристик из событий Activity Hub — автоматический сигнал Аттестатора. Не заменяет Диагноста (cp-срезы §4) — дополняет как baseline-proxy.
>
> **Кодирование:** `bh.` = behaviour (поведенческие метрики из событий). `cp.` = capacity (потенциал через рубрики Диагноста). Старые коды S/T/M/W/A → bh.sys/bh.inv/bh.met/bh.awr/bh.agn.
>
> **Весовая конфигурация:** `DS-ecosystem-development/…/Data-Governance/iwe-actions-catalog.md §7`.

### 12.1 Модель 7 bh-характеристик

| Код | Характеристика | Ось | Тип | Показатель |
|-----|----------------|-----|-----|------------|
| **bh.sys** | Систематичность | Мастерство | обязательная | `self_dev_days_per_week` — среднее дн/нед в скользящем окне |
| **bh.inv** | Инвестиционность времени | Мастерство | обязательная | `avg_hours_per_week` — среднее ч/нед (slot_logged + lesson_completed) |
| **bh.met** | Методичность | Мастерство | компенсаторная | `methodical_events_per_month` — применение методов, событий/мес |
| **bh.awr** | Осведомлённость | Мировоззрение | основная | `worldview_score` — взвешенная сумма событий рефлексии за 30 дней |
| **bh.agn** | Агентность | Мировоззрение | основная | `agency_score` — взвешенная сумма самоинициированных событий |
| **bh.scl** | Масштабность | Мастерство | информационная | `scale_score` — РП с фактическим временем (wp_completed × actual_hours) |
| **bh.stb** | Устойчивость | Мастерство | gate | `max_gap_days` — максимальный разрыв без SELF_DEV активности |

> **Различение типов.** «Обязательная» = ноль блокирует ступень. «Компенсаторная» = низкое не блокирует при компенсации. «Основная» = наибольший вес в оси Мировоззрения. «Gate» = срабатывает как дополнительный порог при расчёте ступени.

### 12.2 Показатели и рубрики

#### bh.sys — Систематичность

**Смысл:** регулярность именно саморазвития. `git_commit`, `wp_created`, `coding_time` — рабочая активность, не считается.

**Принцип окна:** растёт со ступенью. Проактивный оценивается за 24 нед — нужно доказать устойчивость на дистанции.

**Множество событий SELF_DEV_EVENT_TYPES** (см. stage_config.py):
```
lesson_completed, knowledge_extracted, pack_updated, qualification_granted,
day_plan_opened/day_open, day_plan_closed/day_close,
week_plan_closed, month_plan_closed, strategy_session_completed, iwe_session
```

| Индекс | Ступень-аналог | Порог дн/нед | Окно |
|--------|:---:|:---:|:---:|
| **0** | — | < 1 | — |
| **1** | Случайный | ≥ 1 | 1 нед |
| **2** | Практикующий | ≥ 3 | 4 нед |
| **3** | Систематический | ≥ 5 | 8 нед |
| **4** | Дисциплинированный | ≥ 6 | 12 нед |
| **5** | Проактивный | ≥ 6.7 | 24 нед |

> **6.7 дн/нед = 1 день пропуска в месяц.** Достижимо для Проактивного с ежедневной практикой.

#### bh.inv — Инвестиционность времени

**Источники (К4 fix):** только `slot_logged.hours` + `lesson_completed.duration_minutes/60`.

> **К4 (WP-310):** `coding_time` (WakaTime) исключён — он включает все репозитории, в том числе рабочие. Используются только чистые учебные события. Нормативы установлены с учётом занижения этих источников.

| Индекс | Ступень-аналог | Порог ч/нед | Окно |
|--------|:---:|:---:|:---:|
| **0** | — | < 0.5 | — |
| **1** | Случайный | ≥ 0.5 | 1 нед |
| **2** | Практикующий | ≥ 4 | 4 нед |
| **3** | Систематический | ≥ 6 | 8 нед |
| **4** | Дисциплинированный | ≥ 8 | 12 нед |
| **5** | Проактивный | ≥ 10 | 24 нед |

#### bh.met — Методичность

**Показатель:** `COUNT(events за 30 дней)` из: `lesson_completed`, `knowledge_extracted`, `pack_updated`, `qualification_granted`.

| Индекс | Диапазон |
|--------|----------|
| **0** | 0 |
| **1** | 1–3/мес |
| **2** | 4–10/мес |
| **3** | 11–25/мес |
| **4** | 26–50/мес |
| **5** | > 50/мес |

#### bh.awr — Осведомлённость (worldview score)

**Окно:** 30 дней (З4, WP-310 — явно зафиксировано в `AWR_WINDOW_DAYS = 30`).

**Показатель:** взвешенная сумма событий рефлексии за 30 дней → индекс 0–5. Веса: `iwe-actions-catalog.md §7`.

| Индекс | Операционные признаки |
|--------|----------------------|
| **0** | Рефлексий нет |
| **1** | Единичные рефлексии |
| **2** | Начинает видеть паттерны. Системный язык — эпизодически |
| **3** | Регулярные рефлексии (Week Close / Month Close). Различает роли, методы |
| **4** | Чёткий системный язык. Видит себя как деятеля в системах |
| **5** | Формирует системное видение у других |

#### bh.agn — Агентность

**Показатель:** взвешенная сумма за 30 дней → индекс 0–5. Веса: `iwe-actions-catalog.md §7`.

| Индекс | Операционные признаки |
|--------|----------------------|
| **0** | Нет инициированных действий |
| **1** | 1–2 самоинициированных действия/мес |
| **2** | Несколько, есть хотя бы 1 самостоятельный РП |
| **3** | ~50% задач из собственных целей. 1–2 активных РП |
| **4** | Несколько независимых РП. Действует в неопределённости |
| **5** | Формирует повестку для ближнего круга |

#### bh.scl — Масштабность

**Показатель:** `SUM(actual_hours по wp_completed за 30 дней)` → взвешенная сумма → индекс 0–5.

> **К2 fix (WP-310):** bh.scl должен мерить фактически отработанное время по РП (actual_hours из payload wp_completed), а не бюджет заявленный при создании. Без поля `actual_hours` в событии — используется фиксированный вес, что является приближением. Тикет на добавление поля: WP-214 Ф11.

**Информационная характеристика** — не блокирует ступень в текущем MVP. Используется Навигатором для оценки «масштабности деятельности» учеников на ст.3+.

#### bh.stb — Устойчивость

**Показатель:** максимальный разрыв без SELF_DEV события (в днях) за учётный период ступени.

> **bh.stb ≠ bh.sys.** bh.sys = средняя частота (сколько дней из семи). bh.stb = худший разрыв (самый длинный простой). Оба нужны: высокое bh.sys с одним разрывом-3-недели = высокая bh.sys, но низкое bh.stb.

**Нормализация:** `norm_stb(max_gap_days)` из stage_config.py.

| bh.stb индекс | Макс. разрыв | Ступень-аналог |
|:---:|:---:|:---:|
| **5** | ≤ 1 день | Проактивный |
| **4** | 2–3 дня | Дисциплинированный |
| **3** | 4–7 дней | Систематический |
| **2** | 8–14 дней | Практикующий |
| **1** | 15–30 дней | Случайный |
| **0** | > 30 дней | — |

**STB_GATE** (минимальный bh.stb для получения ступени): ст.2 ≥ 1, ст.3 ≥ 2, ст.4 ≥ 3, ст.5 ≥ 4.

| Характеристика | Ст. 1 | Ст. 2 | Ст. 3 | Ст. 4 | Ст. 5 |
|---|:---:|:---:|:---:|:---:|:---:|
| **bh.sys** (обяз.) | — | ≥ 2 | ≥ 3 | ≥ 4 | ≥ 5 |
| **bh.inv** (обяз.) | — | ≥ 2 | ≥ 3 | ≥ 4 | ≥ 5 |
| **Накоп. часы** (hard gate) | — | ≥ 20 ч | ≥ 48 ч | ≥ 96 ч | ≥ 161 ч |
| **bh.met** | — | — | ≥ 1 | ≥ 2 | ≥ 4 |
| **bh.awr** | — | — | ≥ 2 | ≥ 3 | ≥ 4 |
| **bh.agn** | — | — | ≥ 1 | ≥ 2 | ≥ 4 |
| **bh.stb** (gate) | — | ≥ 1 | ≥ 2 | ≥ 3 | ≥ 4 |

> **bh.met ≥ 4 для ст.5** (исправлено WP-310 Ф9: было ≥ 3 — слишком низкий порог для Проактивного, который ежедневно формализует знания).

**Нормализация: bh.sys / bh.inv** — multi-window (берётся наибольший пройденный индекс):

| Индекс | bh.sys: окно | bh.sys: порог | bh.inv: окно | bh.inv: порог |
|:---:|:---:|:---:|:---:|:---:|

| **4** | 12 нед | ≥ 6.0 | 12 нед | ≥ 8.0 |
| **3** | 8 нед  | ≥ 5.0 | 8 нед  | ≥ 6.0 |
| **2** | 4 нед  | ≥ 3.0 | 4 нед  | ≥ 4.0 |
| **1** | 1 нед  | ≥ 1.0 | 1 нед  | ≥ 0.5 |

**Нормализация: bh.met, bh.awr, bh.agn** (взвешенная сумма → индекс):

| Индекс | bh.met: соб/мес | bh.awr: взвеш. | bh.agn: взвеш. |
|:---:|:---:|:---:|:---:|
| **5** | > 50 | > 50 | > 30 |
| **4** | 26–50 | 29–50 | 18–30 |
| **3** | 11–25 | 13–28 | 9–17 |
| **2** | 4–10 | 5–12 | 4–8 |
| **1** | 1–3 | 1–4 | 1–3 |
| **0** | 0 | 0 | 0 |

**Алгоритм расчёта** (см. `stage_config.py::compute_stage_mvp`):

```python
def compute_stage_mvp(s, t, m, w, a, total_hours=0.0, stb=5) -> int:
    """
    s=bh.sys, t=bh.inv, m=bh.met, w=bh.awr, a=bh.agn — индексы 0–5.
    total_hours — накопленные ч (только slot_logged + lesson_completed, К4).
    stb=bh.stb — индекс устойчивости (max-gap).

    Gates в порядке:
    1. Mandatory: s==0 или t==0 → ступень 1
    2. Матрица STAGE_GATE_MATRIX: все 5 bh ≥ минимумов → stage
    3. Hard gate накопленных часов
    4. bh.stb gate
    """
```

> Ступень 5 из этой функции — `bh_gate_passed`. Итоговая ступень 5 требует дополнительно cp-подтверждения (§5.1) и `form093_real_change_gate` (FORM.093 §7.2).

### 12.4 Маппинг bh. → cp. (§4)

### 12.5 Известные ограничения MVP

1. **bh.agn без initiator.** Поле `initiator` в `wp_created` отсутствует (WP-214 Ф10.5). MVP: все wp_created = самоинициированные.
2. **WakaTime опциональный.** Если не подключён — bh.inv опирается только на slot_logged + lesson_completed. Занижение допустимо на ранних волнах.
3. **bh.stb не реализован в rcs-collector.** Нет запроса max_gap в stage_evaluator.py. Добавить в Ф10+ WP-310.
4. **bh.scl без actual_hours.** Фиксированный вес вместо реального времени. Тикет: WP-214 Ф11.
5. **TTL cp-срезов не автоматизирован.** Проверка вручную Диагностом. Автоматический триггер — WP-310 Ф5+.
