# Формализация: Конвейер разработки и изменения руководства

> Описывает жизненный цикл универсального руководства программы ЛР — от концепции до публикации и обновления. Структурно идентичен PD.FORM.100 (конвейер ИТ-программы): каждый этап имеет вход, действие, выход; выход одного — вход следующего; роли с чёткими полномочиями.

## 1. Принцип чтения

Каждый этап: **Вход → Действие (метод/инструмент) → Выход**.
Выход одного этапа становится входом следующего.
Так устроена любая производственная система — не только ИТ.

**Аналогия с ИТ-конвейером (PD.FORM.100):**

| ИТ-конвейер | Конвейер руководства |
|-------------|----------------------|
| PRD (что и зачем) | Концепция руководства (объект, ось, ступени) |
| Архитектура | Структура S→SS + Pack-маппинг |
| Код в ветке | Markdown подраздела |
| Pull Request | Ревью субагентом (🟡) |
| Docker-образ | Скомпилированный подраздел в staging |
| Container Registry | Git-репо с тегами версий |
| Kubernetes | Сайт документации (`aisystant/docs:main`) |
| QA-инженер | Пилот (🟢) с чек-листом 6/6 |
| Production | Прод-сайт для пилотов |
| Мониторинг | Обратная связь пилотов + Pack-drift detector |

## 2. Цепочка (10 этапов)

### Этап 1. Концепция руководства

- **Вход:** запрос (новое руководство нужно) или сигнал обновления (Pack изменился, ступень добавлена, мировоззренческий сдвиг)
- **Действие:** автор определяет объект руководства (Пилот / Машина / инструмент мышления), ось роста (W / M / I), ступени-аудиторию, границы (что НЕ входит)
- **Выход:** концептуальный документ «Что это руководство и почему нужно» (1-2 стр., раздел `## Различение N руководств` в WP-300)

> **Различие с ИТ:** в ИТ источник — бизнес-потребность. Здесь источник — Pack (онтология программы ЛР). Pack-владелец = аналог продакт-менеджера.

---

### Этап 2. Структура руководства (S → SS)

- **Вход:** концепция руководства
- **Действие:** автор раскладывает руководство на разделы (S1...SN) и подразделы (SS1...SSM) с учётом дуги по ступеням; для каждого подраздела фиксирует `mastery_node` (узел), `stage_relevant` (на каких ступенях полезен), `prerequisites`
- **Выход:** `structure-guide-N.md` в `DS-principles-curriculum/specs/v4-reference/` (структура без текста)

> **Артефакт-разделитель:** между концепцией и структурой стоит решение «по чему делить разделы?» — по фазам дуги, по узлам мастерства, по ролям, по методам. Это методическое решение, не техническое.

---

### Этап 3. Pack-маппинг

- **Вход:** структура руководства + Pack PD (FORM/METHOD/CAT/FAIL)
- **Действие:** для каждого подраздела автор указывает `introduces` (новые понятия), `uses` (используемые понятия), `cp_check` / `bh_check` (какие метрики прогресса проверяются), ссылки `(PD.FORM.NNN)` для каждого понятия
- **Выход:** `structure-guide-N.md` с заполненными frontmatter подразделов и evidence-graph

> **Pack — источник истины.** Любое понятие в `introduces` должно быть в Pack. Иначе — gap в Pack (FAIL.NNN) или дрейф.

---

### Этап 3.5. Pack-sufficiency gate (блокирующее)

- **Вход:** структура руководства с `introduces` всех подразделов
- **Действие:** автоматическая проверка покрытия понятий в Pack. Для каждого `introduces`:
  1. Понятие присутствует в `PACK-personal/ontology.md` §2 (канонический реестр).
  2. У понятия есть Pack-источник (`PD.FORM.NNN`, `PD.METHOD.NNN`, `PD.CAT.NNN` или `PD.WP.NNN`).
  3. Омонимы разрешены квалификатором (одно слово с разным `U.*` parent).
- **Выход:** PASS (можно идти к Этапу 4) ИЛИ FAIL c orphan-списком → **«Спринт по Pack, не письмо»**

> **Принцип:** если понятия нет в Pack — его нет в руководстве. Понятие сначала формализуется через Knowledge Extractor (`DP.AISYS.013`), потом попадает в `ontology.md` §2 и Pack-источник, потом становится частью `introduces` подраздела. Обратный порядок (написать текст → потом формализовать) запрещён — он накапливает дрейф и блокирует Портного.

| Сценарий orphan-концепта | Действие |
|---|---|
| Новое понятие в процессе письма | **Стоп.** Зафиксировать в candidates-реестре конвейера (`07-concept-candidates-v4.md`) → Knowledge Extractor → дождаться `ontology.md` §2 → потом письмо. |
| Понятие есть в старом руководстве, но нет в Pack | Старое руководство — legacy, не источник истины. Извлечь → формализовать → потом использовать. |
| Понятие есть в Concept Graph / candidates | Производный артефакт, не источник истины. Ждать формализации. |

**Исключение:** бытовые аналогии и кейсы (не теоретические понятия) допустимы без Pack-источника, но **не попадают** в `introduces` и **не становятся** концепт-нодами графа.

**Аналогия с ИТ-конвейером:** Pack-sufficiency = type-check перед compile. Без неё код «соберётся», но рантайм упадёт у первого же пользователя (Портной не построит маршрут по orphan-концепту).

**Реализация:** `v4-lint.py porter` + `v4-lint.py structure` дают FAIL при orphan-концепте (CHECKLIST-subsection §🔴 A.8). Без PASS — `build-skeleton.py` не создаёт PR-скелет.

---

### Этап 4. Скелет подраздела (детерминированная сборка)

- **Вход:** `subsection_id` + structure + Pack
- **Действие:** автор или скрипт `build-skeleton.py` собирает markdown-каркас: frontmatter + заголовки + плейсхолдеры по структуре, без content
- **Выход:** markdown-файл подраздела со скелетом, готовый к наполнению

> **Цель этапа:** убрать ручную рутину «начать с пустого экрана». Скелет даёт структуру, автор наполняет content.

---

### Этап 5. Наполнение содержимым (рукой автора или LLM Build)

- **Вход:** скелет подраздела + контекст (structure + Pack + role-prefixes + дуга нарратива) + **PACK-rhetoric** (WP-340, когда готов): карточки иллюстраций с тегами по концепту + ступени
- **Действие:** автор пишет текст по WRITING-PIPELINE Этап 4, ИЛИ LLM (Claude API) генерирует draft → автор правит. Применяется онтология «мем → метод → мировоззрение» (см. PD.FORM.087). Иллюстрации (кейсы, аналогии): **сначала проверить PACK-rhetoric** по концепту и ступени — переиспользовать готовое. Если подходящего нет — создать новое и заявить в WP-340 inbox для формализации
- **Выход:** markdown-файл подраздела с заполненным content

> **PACK-rhetoric (WP-340):** единый банк обезличенных иллюстраций — парадейгмы, аналогии, антитезы, тегированные по домену, ступени и концепту. Цель: один концепт объясняется через согласованную аналогию во всех четырёх руководствах, а не через случайно придуманную в моменте.

> **Аналог Docker-образа:** содержимое + структура + метаданные упакованы в один артефакт.

---

### Этап 6. Контроль качества — машинный (🔴 lint)

- **Вход:** заполненный подраздел
- **Действие:** `v4-lint` запускает 5 проверок (structure, porter, cross-guide, pack-drift, graph) — exit 0 = PASS
- **Выход:** PASS (можно дальше) или FAIL (диагностика автору)

> **Аналог CI:** автоматическая проверка структуры, frontmatter, ссылок на Pack, графа понятий. Без понимания content — только формальные правила.

---

### Этап 7. Контроль качества — содержательный (🟡 субагент)

- **Вход:** подраздел, прошедший 🔴 PASS
- **Действие:** субагент Claude Opus читает текст и применяет семантические критерии: границы понятий (A.6), нарративная цепочка (мем→метод→мировоззрение), дуга по ступени (тональность), практика и время, аналогия, связность с Pack
- **Выход:** отчёт PASS/FAIL с цитатами из текста

> **Аналог Pull Request review:** другой носитель смотрит на работу и решает «можно мержить или вернуть».

---

### Этап 8. Доставка в staging

- **Вход:** подраздел, прошедший 🔴+🟡 PASS
- **Действие:** CD pipeline (GitHub Actions) копирует файл в `aisystant/docs:staging-v4/docs/ru/personal-new-staging/`, пересобирает сайт документации, публикует по URL
- **Выход:** staging-версия подраздела, доступная по URL для пилот-теста

> **Аналог staging-кластера Kubernetes:** боевая инфраструктура, но изолированная от прод-пользователей.

---

### Этап 9. Пилот-тест (🟢 живой человек)

- **Вход:** staging-версия подраздела
- **Действие:** ≥3 пилотов читают подраздел, отвечают на 6 критериев (мем снят, метод применим, время соответствует, аналогия помогает, нет барьеров, мировоззренческий сдвиг) через issue-шаблон `pilot-feedback`
- **Выход:** label `pilot-approved` (≥3 пилота × 6/6) ИЛИ `needs-rework` (возврат на этап 5)

> **Аналог QA-инженера:** проверка работает ли продукт у реального пользователя, не в лабораторных условиях.

---

### Этап 10. Деплой в production + мониторинг

- **Вход:** подраздел с label `pilot-approved`
- **Действие:** merge `staging-v4` → `main`, CD pipeline деплоит в `aisystant/docs/docs/ru/personal-new/`, TG-уведомление активным пилотам; параллельно работает Pack-watcher — следит за изменениями Pack и создаёт issue при drift > 0
- **Выход:** подраздел в прод-руководстве + обратная связь пилотов + drift-issue при изменении Pack → вход следующего цикла обновления (→ этап 1 или этап 5)

> **Аналог Prometheus/Grafana:** непрерывное наблюдение за состоянием системы. Сбой = сигнал, не катастрофа.

## 5. Системное различение

Конвейер руководства — **методическая производственная система** (уровень 3 по PD.FORM.027: социо-технические системы, потому что роль пилота и автора — живые носители).

Каждый этап — роль с чётким входом и выходом. Нет лишних шагов.
Сбой на любом этапе не пропускается вперёд — это принцип fail-fast:
- FAIL 🔴 → возврат к этапу 5 с диагностикой
- FAIL 🟡 → возврат к этапу 5 с замечаниями субагента
- FAIL 🟢 → возврат к этапу 5 с label `needs-rework`

## 6. Различия с PD.FORM.100 (ИТ-конвейер)

| Аспект | PD.FORM.100 (ИТ) | PD.FORM.103 (Руководство) |
|--------|------------------|----------------------------|
| **Источник** | Бизнес-потребность пользователя | Pack PD (онтология) |
| **Артефакт** | Docker-образ (исполняемый код) | Markdown подраздел (текст) |
| **QA** | E2E + smoke тесты | 6 критериев пилот-теста |
| **Контроль качества** | Только 🔴 (lint, тесты) | 🔴+🟡+🟢 (машина+агент+пилот) |
| **Мониторинг** | Prometheus (метрики работы) | Pack-drift + обратная связь пилотов |
| **Деплой** | Kubernetes (rolling update) | Git push в `aisystant/docs:main` |
| **Версионирование** | Semver `v1.2.0` | Semver руководства `v4.X.Y` |

**Главное различие:** в ИТ можно проверить «работает / не работает» машинно. В методической работе значительная часть проверки требует человека (Этап 7 — субагент, Этап 9 — пилот). Это не недостаток, а свойство домена.

## 7. Применение в программе ЛР

Используется в трёх режимах:

1. **Создание нового руководства** — полный проход 1→10
2. **Обновление подраздела** — старт с этапа 5 (содержимое), далее 6→10
3. **Drift-реакция** — старт с этапа 10 (получено drift-issue) → этап 1 (концептуальное решение: нужно обновлять или нет?) → 5→10

Контекст: руководства программы ЛР (WP-300), конвейер платформы (WP-322), Pack-владение (PACK-personal).
