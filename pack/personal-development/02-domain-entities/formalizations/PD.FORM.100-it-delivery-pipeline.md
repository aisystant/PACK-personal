# Формализация: Конвейер доставки ИТ-программы

> Используется в программе ЛР как конкретный пример системы с входами–выходами и чёткими ролями на каждом этапе. Даёт ИТ-контекст для практики системного мышления.

## 1. Принцип чтения

Каждый этап: **Вход → Действие (метод/инструмент) → Выход**.
Выход одного этапа становится входом следующего.
Так устроена любая производственная система — не только ИТ.

## 2. Цепочка

### Этап 1. Формулировка потребности
- **Вход:** бизнес-потребность / проблема пользователя
- **Действие:** продакт-менеджер проводит интервью, строит user story map
- **Выход:** PRD (Product Requirements Document) — описание «что» и «зачем»

---

### Этап 2. Проектирование
- **Вход:** PRD
- **Действие:** архитектор / тех. лид применяет C4-модель, ADR
- **Выход:** техническое задание + архитектурное решение (сервисы, БД, API)

---

### Этап 3. Написание кода
- **Вход:** ТЗ + архитектурное решение
- **Действие:** разработчик пишет код в ветке `feature/...`, покрывает unit-тестами
- **Выход:** код в ветке

---

### Этап 4. Контроль качества кода — Pull Request
- **Вход:** код в ветке
- **Действие:** коллеги делают code review — проверяют логику, безопасность, стиль
- **Выход:** одобренный PR, влитый в `main`

> **Pull Request (PR)** — это не кнопка «залить», а контрольная точка: код не попадает в основную ветку, пока его не одобрили другие.

---

### Этап 5. Упаковка в образ — Docker build
- **Вход:** коммит в `main`
- **Действие:** CI Pipeline запускает `docker build` — упаковывает код вместе со всеми зависимостями
- **Выход:** Docker-образ (артефакт: операционная система + runtime + код в одном контейнере)

> **Docker** решает проблему «у меня работает, а на сервере нет»: образ содержит всё необходимое и одинаково запускается на любой машине.

---

### Этап 6. Проверка и сохранение образа
- **Вход:** Docker-образ
- **Действие:** CI Pipeline запускает тесты, линтер, проверки безопасности внутри образа; затем `docker push`
- **Выход:** проверенный образ загружен в Container Registry с тегом версии (`v1.2.0`)

> **Container Registry** — хранилище версий образов. Как Git для кода, только для упакованных программ.

---

### Этап 7. Доставка в тестовую среду — деплой в staging
- **Вход:** образ в Registry
- **Действие:** CD Pipeline (ArgoCD / Helm) передаёт новый тег образа в **Kubernetes**
- **Выход:** Kubernetes запускает Pod-ы с новой версией, делает rolling update без остановки сервиса

> **Kubernetes** — платформа запуска контейнеров. Берёт образ и запускает нужное количество копий (Pod-ов), следит за их здоровьем, перезапускает упавшие. **Деплой** — действие передачи нового образа в Kubernetes.

---

### Этап 8. Тестирование
- **Вход:** новая версия запущена в Kubernetes (среда staging)
- **Действие:** QA-инженер проводит e2e-тесты, smoke-тесты, ручную проверку
- **Выход:** «готово к проду» или список дефектов → возврат к этапу 3

---

### Этап 9. Деплой в production
- **Вход:** подтверждение QA
- **Действие:** CD Pipeline деплоит тот же образ в production-среду Kubernetes
- **Выход:** программа работает у клиента

---

### Этап 10. Мониторинг
- **Вход:** работающая программа в production
- **Действие:** Prometheus, Grafana, Sentry собирают метрики и алерты непрерывно
- **Выход:** данные о здоровье системы → вход для следующего цикла (→ этап 1)

---

## 3. Карта инструментов

| Инструмент | Роль в конвейере |
|------------|-----------------|
| **Git / PR** | Контроль качества кода до слияния (этап 4) |
| **Docker** | Упаковка кода в переносимый артефакт (этапы 5–6) |
| **Container Registry** | Хранение версий образов (этап 6) |
| **CI Pipeline** | Автоматическая сборка и проверка (этапы 5–6) |
| **CD Pipeline** | Автоматическая доставка образа (этапы 7, 9) |
| **Kubernetes** | Платформа запуска, масштабирования и самовосстановления (этапы 7, 9) |
| **Мониторинг** | Видимость: работает ли у клиента (этап 10) |

## 4. Системное различение

Конвейер — это **производственная система** (уровень 2 по PD.FORM.027: кибер-физические системы).
Каждый этап — роль с чётким входом и выходом. Нет лишних шагов.
Сбой на любом этапе не пропускается вперёд — это принцип fail-fast.

Применение в ЛР: использовать как модель при анализе любого конвейера (обучение, публикация, исследование).
