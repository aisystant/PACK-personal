# PD.METHOD.051 — «Блюдо, не кухня»: объяснение через результат, а не инфраструктуру

## Описание

При объяснении платформенных или системных концепций широкой аудитории — показывать только конечный результат (что человек получает), не механизм его производства. «Кухня» (ингредиенты, источники, слои, инфраструктура) скрыта; «блюдо» (готовый продукт взаимодействия) — на переднем плане.

## Forces

_(Optional, WP-448 Ф12) Какие конкурирующие давления удерживает метод._

| Force | Tension |
|-------|---------|
| Прозрачность механизма ↔ фокус на результате | IPO шаг 2 предписывает убирать деталь в «кухню», даже если она честно объясняет, откуда берётся результат — метод жертвует прозрачностью ради нерассеянного внимания на «блюде» |
| Экспертная точность ↔ доступность неэксперту | Полное описание слоёв (Pack, FPF — см. Пример WP-442) точнее для эксперта, но именно эти слои метод предписывает прятать, потому что аудитория неэкспертная |
| Доверие через показ источников ↔ простота решения о пользе | Тест применимости («нужна ли деталь, чтобы принять решение о пользе?») сознательно отсекает информацию об источниках/«рынке», которая могла бы повысить доверие, но перегружает решение |

## IPO

**Input:** объяснение системы/платформы + широкая (неэкспертная) аудитория

**Process:**
1. Для каждой детали объяснения задать тест: «нужна ли аудитории эта деталь, чтобы принять решение о пользе?»
2. Если нет → убрать деталь в «кухню» (не показывать)
3. Если да → оставить в «блюде» с конкретным примером
4. Проверить: текст объясняет только конечный результат взаимодействия?

**Output:** объяснение, сфокусированное на результате для потребителя, без видимой инфраструктуры

## Тест применимости

«Нужна ли аудитории эта деталь, чтобы принять решение о пользе?» Нет → убрать.

## Пример

WP-442 C-032: первоначальный текст показывал «кухню» (Pack, FPF, ингредиенты: соль, мука) и «рынок» (откуда берутся ингредиенты). После применения метода — только «блюдо»: что именно человек получает при взаимодействии с системой. Итерации: Pack/FPF → поваренная книга → кухня убрана → конкретика вместо абстракций.

## Применимость

Любое объяснение платформ, систем или методологий для аудитории, которой нужен результат, а не архитектура.

## Bias-Annotation

_(Optional, WP-448 Ф12) Куда систематически съезжает внимание практикующего. Не дублирует «Failure Modes» — там «что пойдёт не так», здесь «куда дрейфует внимание»._

| Bias | Direction of distortion |
|------|--------------------------|
| Тест подменяется деталью | Внимание съезжает с вопроса теста применимости («нужна ли деталь для решения о пользе?») на «насколько подробно я могу объяснить механизм» — авторская гордость за инфраструктуру тянет обратно к «кухне» |
| Метафора подменяет проверку | При работе с примером (итерации WP-442) внимание смещается на поиск ярких метафор («блюдо», «кухня»), а не на прогон каждой детали через тест применимости — метафора замещает саму проверку |
