# [PD.METHOD.031] FORM Slices + Node Tree

## Назначение

Организационный принцип модели развития созидателя: декомпозиция через двойную структуру — Node Tree (дерево узлов, что развивается) + Slices (срезы, как это измеряется в конкретный момент).

## Два ортогональных взгляда

### Node Tree (Дерево узлов)

Дерево узлов отвечает на вопрос **«что развивается»** у созидателя:

```
Созидатель
├── Мастерство (M)
│   ├── M1 — Систематичность (ритм + методы)
│   ├── M2 — Экзокортекс (IWE + ОРЗ)
│   ├── M3 — Инвестированное время
│   └── M4 — Агентность (практические рабочие продукты)
├── Мировоззрение (W) — зрелость, не наличие
└── Интеграция (IT) — связность практики с реальными задачами
```

Узел дерева = то, что можно развивать методами (M1-M4, W, IT). Агентность (A) = gate-индикатор, не узел.

### Slices (Срезы)

Срезы отвечают на вопрос **«как это измеряется здесь и сейчас»** — диагностический инструмент для конкретной оси в конкретный момент:

- Срез cp.rhy → измеряет M1 (ритм)
- Срез cp.wld → измеряет W (зрелость мировоззрения)
- Срез cp.skl → измеряет M1-M4 (уровень «умею»)
- Срез cp.iwe → измеряет M2 (Экзокортекс)
- Срез cp.int → измеряет IT (интеграция)
- Срез cp.agt → измеряет A (агентность как gate)

## Forces

_(Optional, WP-448 Ф12) Какие конкурирующие давления удерживает метод._

| Force | Tension |
|-------|---------|
| Дерево узлов (что развивается) ↔ Срезы (как измеряется) | Диагност (R28) видит только срезы (cp.rhy, cp.wld...) и проецирует их на дерево узлов, чтобы найти bottleneck-узел («Использование в диагностике»); проектировщик руководства должен пройти обратный путь — от узла к срезам, иначе получаются «главы о срезе», а не «главы о развиваемом качестве» («Использование в curriculum design») |
| Единый объект ↔ разные роли узла и среза | Один и тот же объект (например W) может быть и узлом развития, и срезом измерения одновременно («Срез ≠ Узел», п.3 Принципа применения) — метод удерживает это раздвоение, не позволяя дизайнеру подменить одну роль другой |
| Агентность как gate ↔ M1-M4/W/IT как узлы | Дерево узлов явно исключает Агентность (A) из узлов развития («A = gate-индикатор, не узел»), но программа одновременно развивает узлы и проверяет готовность через A — два разных режима работы с одним и тем же деревом |

## Принцип применения

1. **Node Tree** = что проектировать (программа, руководство, методы)
2. **Slices** = что измерять (диагностика, прогресс, переход ступеней)
3. **Срез ≠ Узел** — один и тот же объект (W) может быть и узлом развития, и срезом измерения, но роли разные (что развивается vs как измеряется)

## Использование в диагностике (R28)

Диагност видит срезы (cp.rhy, cp.wld...) → проецирует на Node Tree → определяет bottleneck-узел → Навигатор R27 выбирает метод для этого узла.

## Использование в curriculum design

При проектировании руководства — организовывать не по срезам, а по узлам. Срез — измерительный инструмент; узел — то, чему учим. Смешение: пишем «главу о cp.rhy» вместо «главы о Систематичности» → дидактически непрозрачно.

_(Optional, WP-448 Ф12) Куда систематически съезжает внимание практикующего. Не дублирует «Failure Modes» — там «что пойдёт не так», здесь «куда дрейфует внимание»._

Tentative — карточка в статусе draft, ещё не откалибрована практикой:

| Bias | Direction of distortion |
|------|--------------------------|
| Срез подменяет узел при проектировании | При написании руководства внимание смещается на готовый, конкретный ярлык среза (cp.rhy, cp.wld...), а не на узел, который он измеряет — отсюда риск, прямо названный в «curriculum design»: «глава о cp.rhy» вместо «главы о Систематичности» |
| Роль объекта «застревает» на последнем использовании | Когда объект (например W) только что использован R28 как срез в диагностике, внимание при последующем проектировании руководства продолжает трактовать его как срез, а не переключается обратно на роль узла — различение «Срез ≠ Узел» неявно теряется |
