# [PD.METHOD.036] Session Time Accounting

## Определение

Метод фиксации факта работы по сессии — специализация [PD.METHOD.001](./PD.METHOD.001-time-accounting.md) для уровня одной сессии (~30 мин — 3ч непрерывной работы по одному РП). Включает два момента регистрации: **оценку бюджета до начала** и **факт по итогам**, оба привязанные к идентификатору РП.

Метод НЕ предписывает длительность сессии, частоту перерывов, технику фокусировки — это смежные методы (PD.METHOD.023 управление потоком).

## Назначение

| Функция | Описание |
|---------|----------|
| **Калибровка оценок** | Сравнение «оценка до» с «факт после» создаёт обратную связь для будущих оценок |
| **Привязка времени к РП** | WakaTime не различает «работа по РП X» от «всё подряд» — ручная фиксация устраняет пробел |
| **Накопительный учёт** | Сумма факта по сессиям одного РП = реальный бюджет РП по итогам |
| **Источник мультипликатора** | См. [PD.FORM.104](../02-domain-entities/formalizations/PD.FORM.104-iwe-multiplier.md) §6 (ручной мультипликатор) |

## Forces

_(Optional, WP-448 Ф12) Какие конкурирующие давления удерживает метод._

| Force | Tension |
|-------|---------|
| Оценка до ↔ факт после (без искажения) | «Калибровка оценок» строится на сравнении «оценка до» с «фактом после», но регистрация оценки перед началом создаёт риск заякоривания: факт при закрытии сессии может неосознанно подгоняться под уже названную оценку вместо независимой регистрации |
| Гранулярность сессии (≥30 мин по одному РП) ↔ многозадачность реальной работы | Метод требует один РП-ID на сессию, но в разделе «Когда не применим» прямо назван случай «многозадачность с быстрым переключением между РП (<15 мин на каждый)» — структура сессии не покрывает реальный режим переключения |
| Факт сессии (декларация пилота) ↔ WakaTime (машинный замер) | Различение «Факт сессии ≠ WakaTime» фиксирует два независимых источника учётного времени за один день; Момент 3 требует сверять их расхождение как сигнал, а не выбирать один источник как единственно верный |

## Протокол

### Момент 1: Открытие сессии

Регистрация перед началом работы:

| Поле | Описание | Обязательно? |
|------|----------|--------------|
| РП-ID | Идентификатор рабочего продукта (WP-NNN или эквивалент) | Да |
| Оценка длительности | Сколько планируется потратить, в часах (0.5h gran.) | Да |
| Ожидаемый результат | Артефакт или промежуточное состояние | Да |
| Метод | Какой метод применяется (если известен) | Нет |

Если РП-ID не определён до начала — сессия не считается работой по методу. Возможны два пути: (a) определить РП-ID до начала; (b) пометить сессию как «незапланированная» и работать по [протоколу открытия](../../../../memory/protocol-open.md) WP Gate.

### Момент 2: Закрытие сессии

Регистрация по итогам:

| Поле | Описание | Обязательно? |
|------|----------|--------------|
| РП-ID | Тот же, что при открытии | Да |
| Факт длительности | Реальное время работы, в часах | Да |
| Достигнутый результат | Что вышло на самом деле (артефакт / состояние) | Да |
| Расхождение | Если факт ≠ оценка на >20%: причина (1 строка) | Условно |

Расхождение >20% без причины — индикатор низкого качества оценки. Серия таких расхождений по одному типу РП → сигнал откалибровать базовую оценку для этого типа.

### Момент 3: Закрытие дня

Агрегация сессий дня:

- `Бюджет закрыт` = `Σ бюджет(РП.status=closed во время дня)` — числитель мультипликатора PD.FORM.104.
- `Сумма факта сессий` = `Σ факт(сессии дня)` — независимый от WakaTime источник учётного времени.
- Расхождение между `Сумма факта` и `WakaTime` за день — сигнал: либо не все сессии зарегистрированы, либо WakaTime не покрывает (не-кодинговая работа).

## Различения

| Различение | Почему важно |
|------------|--------------|
| Сессия ≠ помодоро | Сессия = единица учёта работы по РП. Помодоро = техника фокусировки (25/5 мин). Помодоро может быть внутри сессии. |
| Оценка ≠ план | Оценка регистрирует ожидание длительности. План регистрирует намерение сделать. Можно иметь план без оценки, но не оценку без РП. |
| Факт сессии ≠ WakaTime | Факт — декларация пилота, привязана к РП. WakaTime — машинный замер времени у клавиатуры без привязки. См. также [DP.FM.036](../../../../PACK-digital-platform/pack/digital-platform/05-failure-modes/DP.FM.036-wakatime-coding-time-bias.md). |

## Bias-Annotation

_(Optional, WP-448 Ф12) Куда систематически съезжает внимание практикующего. Не дублирует «Failure Modes» — там «что пойдёт не так», здесь «куда дрейфует внимание»._

| Bias | Direction of distortion |
|------|--------------------------|
| Расхождение оправдывается вместо накопления как сигнал | При факт≠оценка >20% внимание уходит на заполнение поля «причина» как разового объяснения конкретной сессии, а не на отслеживание серии расхождений одного типа РП, которая по методу должна вести к пересмотру базовой оценки |
| Отсутствие РП-ID закрывается задним числом, а не по протоколу | Когда РП-ID не определён до начала, внимание съезжает на то, чтобы начать работу и приписать ID позже, минуя оба явно названных пути метода — определить ID заранее или пометить сессию «незапланированной» через WP Gate |

## Failure Modes

- [PD.FAIL.002](../05-failure-modes/PD.FAIL.002-time-accounting-is-discipline.md) — путать фиксацию факта с дисциплиной. Метод регистрирует, не предписывает.

## Применимость

**Когда применим:**
- Работа разбивается на дискретные сессии длиной ≥30 мин по одному РП
- Есть инфраструктура для записи (DayPlan, фиксация в чате, лог)
- Оценка качества метода времени важна (нужна калибровка)

**Когда не применим:**
- Длинная непрерывная работа без естественных границ
- Многозадачность с быстрым переключением между РП (<15 мин на каждый)
- Поисковая, исследовательская работа без чёткого ожидаемого результата
