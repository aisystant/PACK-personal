# Метод расчёта мультипликатора IWE

> **Рамка (читать первой).** Мультипликатор — это **прибор диагностики узкого места**, а не оценка успеха. Как только число становится целью, его начинают накручивать (закон Гудхарта; эмпирически подтверждено DORA: оптимизация под скорость письма ломает стабильность доставки). Поэтому метод описывает **не одну формулу, а панель из пяти показателей**, где множитель — лишь один тайл, всегда с подписью числа агентов и без сравнения между периодами с разной методикой. Пятый тайл — когнитивная нагрузка — показывает цену рычага: усталость от решений (PD.SOTA.003), без неё высокий множитель выглядит здоровым, даже когда куплен ценой выгорания пилота.
>
> Источник: пир-сессия Claude + Kimi 2026-06-11 (DP.SC.154), грунт-бриф из 21-агентного исследования (METR 2025, DORA 2024-2025, SPACE/DevEx, EVM, Goldratt, Goodhart/Campbell, Kahneman/Flyvbjerg).

## 1. Ядро

```
Множитель(период) = Σ_РП(засчитанный бюджет закрытого РП) / Инвестированное физ-время(период)
```

Прямой отраслевой аналог — индекс освоенного объёма CPI (Cost Performance Index) из управления проектами (US DoD, 1960-е): `CPI = освоенная оценка / фактические затраты`. Читается зеркально (у CPI хорошо = ≥1 при недорасходе, у нашего множителя — рост выработки на час), но математически родственен. Наследуем и его уязвимость (числитель оценивает та же сторона), и его лекарство (бюджет фиксируется в baseline **до** работы независимой стороной).

<details open>
<summary><b>2. Числитель — засчитанный бюджет</b></summary>

**2.1. Заморозка бюджета при открытии (freeze-at-assignment).** Бюджет РП фиксируется в момент согласования на WP Gate (там уже требуется оценка ~Xh + согласие пилота = готовый baseline + независимая сторона). **Пересмотр бюджета задним числом при закрытии запрещён.** Это гасит накрутку «было 3, стало 8» (инфляция оценок как у story points).

**2.2. Оценка через reference-class, не «сколько бы заняло без ИИ».** Последняя фраза — якорь: он раздувает оценку подсознательно даже когда сказано его игнорировать (Jorgensen 2004). Убрать из процесса оценки структурно; контрфакт «без ИИ» оставить только для нарратива пилоту, не для числителя. Вместо него: бюджет = медиана факта похожего класса прошлых РП. **Готовый класс — классификатор верификации:** trivial / closed-loop / open-loop / problem-framing.

**2.3. Калибровка по факту (bias-factor).** По каждому закрытому РП хранить пару (оценка, физ-факт). Личное смещение = медиана отношения (оценка / факт). Систематически завышаешь → будущие оценки делятся на это смещение. Это автоштраф: раздул прошлые оценки → будущие режутся сильнее.

**2.4. Трёхточечная оценка для рискованных.** Для open-loop / problem-framing — оценка из трёх точек (оптимистичная O, вероятная M, пессимистичная P): `E = (O + 4M + P) / 6`. Обязательный пессимистичный P структурно мешает занизить и даёт честный диапазон.

</details>

<details>
<summary><b>3. Знаменатель — инвестированное физ-время</b></summary>

**3.1. Граница знаменателя зафиксирована (решение пилота: узкая граница).** Берётся **только время в редакторе** (WakaTime), как сейчас. Это инвестированное в РП время (различение учтённое / инвестированное / потраченное → берём инвестированное).

**3.2. Известное завышение принято осознанно.** WakaTime не считает диалог с агентом, проектирование, ожидание ответа модели и режет простои → знаменатель занижен → множитель систематически **завышен**. Решение пилота: не расширять захват сейчас (это потребовало бы нового инструмента), а **держать границу неизменной между неделями**. Следствие-инвариант: множитель сопоставим только сам с собой во времени, не с внешними бенчмарками. Расширение захвата — отдельная возможная задача.

**3.3. Знаменатель = wall-clock пилота, аннотированный числом агентов N.** Параллельность **легитимно** поднимает множитель (3 агента в один час пилота = тройной рычаг экзоскелета). Это не баг, а сам диагностический сигнал: множитель растёт с N до насыщения надзора (≈4-6 параллельных, PD.SOTA.003), затем плато. Плато показывает личный потолок параллельности. Принцип PD.SOTA.003 «множитель ≠ f(N)» означает «нельзя получить безграничный множитель добавлением агентов» (надзор насыщается сам), а НЕ «убрать N из формулы». Поэтому рядом с числом всегда подпись: при скольких агентах оно получено. Период с N=4 и период с N=1 не сравниваются в лоб.

</details>

<details>
<summary><b>4. Незакрытые РП — учёт по Goldratt + дробление (решение пилота)</b></summary>

**4.1. Default — только закрытое (Throughput Accounting, Goldratt).** Незакрытый РП = замороженный капитал (Inventory), не прогресс. В числитель идёт только полностью закрытое. Незакрытый РП → 0 — это правильное поведение, не баг.

**4.2. Многонедельный РП дробится на закрываемые атомы (rolling WP Gate).** Долгий РП = цепочка недельных под-РП / карточек задач, каждая закрывается отдельно с собственным бюджетом и гейтом. Прогресс виден через «закрыто X из N карточек», а числитель остаётся чист. Совпадает с различением IWE «карточка задачи (атом конвейера) ≠ РП (зонт работы)» (РП403): атом учёта = карточка, не зонт.

**4.3. Узкое исключение — partial credit для неделимого problem-framing.** Только для исследовательского РП, где результат есть лишь в конце и дробление на недельные атомы неестественно. Тогда — взвешенные вехи (weighted-milestone): при открытии РП разбить на фазы (frontmatter `phases`), зафиксировать вес каждой (Σ=1) и **объективный гейт приёмки** (артефакт / коммит / проверка прошла). Засчитывать `budget × Σ(веса пройденных гейтов)`. Внутри фазы partial credit нет («половина сваи ≠ половина сваи»), кредит появляется только при закрытии гейта. Веса заморожены при открытии. **До закрытия такой РП штрафуется по возрасту незавершёнки (WIP-age)** — чтобы не плодить «вечные 90%».

**4.4. Веса фаз — свойство класса, не РП (решение по В4).** Веса назначает классификатор верификации из медианы истории класса при открытии (заморожены). Пилот может обжаловать, default = frozen reference-class. Это закрывает self-reported канал Гудхарта без трения на каждом открытии.

</details>

<details>
<summary><b>5. Панель из пяти показателей (вместо одного числа)</b></summary>

Вся современная наука (SPACE, DevEx) отказалась от единственной метрики продуктивности: «одно число слишком легко накрутить». Поэтому итог — панель, накрутить сразу 5 разнонаправленных чисел без реального улучшения нельзя:

| Тайл | Что показывает | Источник-аналог |
|------|----------------|-----------------|
| **Рычаг (Leverage)** | Множитель из §1, с подписью N и флагом «несравнимо при разном N или границе захвата» | EVM CPI |
| **Поток (Delivery Rate)** | Число закрытых РП / период | Throughput (Goldratt), #NoEstimates — неподделываем, не зависит от оценки |
| **Переделка (Rework Rate)** | Доля переоткрытых РП от всех закрытых | DORA change failure rate |
| **Точность оценки (Estimation Bias)** | \|план − факт\| / факт | калибровка bias-factor |
| **Когнитивная нагрузка (Decision Load)** | Σ весов решений пользователя за период / порог | SPACE «благополучие», Sustainability metrics (PD.SOTA.003 §5) |

Заголовочного «балла успеха» над панелью нет.

### 5.1. Тайл когнитивной нагрузки — детали

Это **цена рычага**: множитель показывает, сколько работы выдал экзоскелет, а нагрузка — сколько решений это стоило пилоту. Высокий множитель, купленный лавиной тяжёлых решений на пилота, неустойчив (усталость от решений = bottleneck PD.SOTA.003: исполнение → надзор → decision fatigue).

**Источник данных — реестр решений (Decision Register, PD.SOTA.003→РП109).** На каждом рубеже фиксируется **только решение пользователя** (не агента — тест §2a: «сделал бы агент сам? да → не писать») с когнитивным весом:

| Тип решения | Вес |
|-------------|-----|
| Подтверждение нетривиального предложения (`approve`) | 1 |
| Отклонение / переделка (`reject`) | 3 |
| Перенаправление задачи (`redirect`) | 3 |
| Выбор архитектуры (`architectural`) | 5 |
| Решение о приоритетах / scope / бюджете (`strategic`) | 5 |

**Показатель = Σ(вес) за период** относительно дневного порога из конфига дня (`day-rhythm-config.yaml`; ориентир из предложения PD.SOTA.003 — ~50 единиц/день, предупреждение при 70%; число не зашито, калибруется). Записи: файл `decisions/decision-log-YYYY-MM.md` + база `user_events`.

**Как читать в связке:** рычаг ↑ при нагрузке в норме = здоровый рост; рычаг ↑ при нагрузке у порога = рост за счёт перегруза пилота, неустойчиво → сигнал снизить N параллельных или уйти в closed-loop сессии. Это тот самый качественный слой (decision fatigue), который множитель по бюджет/время сам не видит.

**Состояние данных (честно):** правило записи и схема весов работают (§2a `protocol-work.md`), логи решений ведутся вручную на рубежах. НЕ подключено: порог `daily_decision_points` в конфиге дня и автоматическая агрегация Σ(вес) из базы — это часть до-вязки (см. §8). **Оговорка-риск:** пока лог ведётся вручную, тайл сам подвержен недосчёту (пропущенный рубеж → нагрузка занижена → рычаг выглядит здоровее, чем есть) — тот же класс ошибки, что недосчёт пир-сессий в §6.2. До автоматизации читать тайл как нижнюю границу нагрузки.

</details>

<details>
<summary><b>6. Защита от накрутки и пропусков</b></summary>

Защита отвечает на конкретные механизмы поломки прокси-метрики (таксономия MIRI): раздувание бюджета = adversarial-Гудхарт (агент искусственно коррелирует прокси с наградой), рост числа без роста ценности = extremal/regressional-Гудхарт.

**6.1. Считает не агент-исполнитель (trusted reference).** Множитель вычисляет НЕ тот агент, что его оптимизирует. Read-only калькулятор с зафиксированным хэшем; данные из независимых источников: бюджет — из контекста РП на момент открытия, время — из WakaTime API напрямую, не из самоотчёта агента. Эмпирика: при доступе к расчётному коду ИИ-агент пытается подделать оценку ~50% эпизодов (RewardHackingAgents 2026). Проверку возложить на изолированного агента (Haiku / VR.R.001), не считавшего множитель.

**6.2. Чеклист полноты (аудит):**
1. **Сверка суммы (control total):** Σ часов всех сессий недели = независимый итог WakaTime.
2. **Непрерывность (sequence check):** все session_id на месте, нет пропущенных сессий. Лечит известный недосчёт пир-сессий (занижал в 2-3 раза).
3. **Дедупликация:** каждый РП в числителе ровно раз (переоткрытый не добавляет бюджет повторно); время одной сессии не дублируется между параллельными РП.
4. **Аудит-трейл (auditability):** к любой цифре путь до первички — список РП с бюджетами и коммитами, лог сессий, выгрузка WakaTime. Пилот может проследить любое число.

</details>

<details>
<summary><b>7. Период раскачки (bootstrap) по классам</b></summary>

bias-factor и reference-class требуют чистой истории. На малой выборке медиана = шум под видом числа. Поэтому по каждому классу верификации:

| История класса | Что показываем |
|----------------|----------------|
| **N < 5 закрытых РП** | Множитель НЕ показывается. Только сырые показатели (поток, переделка). Плашка «калибровка недостаточна». |
| **5 ≤ N < 15** | Интервал, не точка: «2.1 [1.2–3.8]» (по разбросу истории класса). |
| **N ≥ 15** | Точечная оценка с поправкой на смещение. |

N считается на класс. Редкие problem-framing РП могут долго жить в режиме «только сырые показатели» — это честно: для них множитель и не должен претендовать на точность.

</details>

## 8. Связь с текущей практикой и что менять дальше

Де-факто формула сейчас — `Σ(бюджет закрытых РП) / WakaTime` в недельном/дневном расчёте, с правилами в памяти (бюджет ≠ физ-время; факт, не план; partial = % × бюджет; мелкие 0.25h; считать все пир-сессии). Новый метод **уточняет**, а не отменяет суть:

- **Меняется:** subjective «% выполнения» → Goldratt + дробление (узкое исключение — взвешенные вехи); одно число → панель из 5; добавлены заморозка бюджета, reference-class, bias-factor, bootstrap, trusted reference, аннотация N, тайл когнитивной нагрузки.
- **Остаётся:** факт-а-не-план, единый недельный расчёт, чеклист полноты (бывшее «считать все пир-сессии»), реестр решений и схема весов (§2a `protocol-work.md`).
- **Реализация в DS-my-strategy (вне Pack):** (1) врезать панель и заморозку бюджета в скрипты недельного/дневного расчёта (`scripts/week-draft-init.sh`, `scripts/day-open-scaffold.sh`); (2) обновить Правила 8-9 в `memory/feedback_governance.md`; (3) **до-вязать тайл нагрузки** — вернуть порог `daily_decision_points` в `day-rhythm-config.yaml` и подключить автоматическую агрегацию Σ(вес) решений из базы `user_events` (ветка БД была сломана в PD.SOTA.003 Ф7, хук захвата Ф7b отложен); (4) ревизовать пороги в PD.SOTA.003 по факту обкатки.
